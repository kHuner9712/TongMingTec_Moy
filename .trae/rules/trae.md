本项目名称：MOY（墨言）
公司主体：桐鸣科技
项目类型：企业级 AI 原生 Web 系统
当前执行文档体系：SSOT
你在本项目中的首要职责：严格按 SSOT 文档执行开发，保证代码、测试、文档长期一致。

========================
一、文档权威规则
========================

1. 唯一执行入口

- 本项目唯一执行入口是：docs/SSOT/
- 任何新任务、新实现、新修订，都必须以 docs/SSOT 为准
- docs/P0、docs/FINAL、docs/\_archive 只允许作为历史背景参考，不允许作为当前实现依据
- 如果历史文档与 SSOT 冲突，一律以 SSOT 为准

2. 文档读取优先级
   每次新任务启动时，先按以下顺序读取文档：

- 00*README*唯一执行入口.md
- 01\_产品范围与阶段地图.md
- 03\_全量需求总表.md
- 04\_需求追踪矩阵.md

然后按任务类型读取事实归属文档：

- 数据库/对象/迁移：05*对象模型与数据库设计.md + 05A*终局关键表DDL展开.md
- 状态流转：06\_状态机总表.md
- 权限/数据范围/AI 边界：07\_权限模型与AI边界.md
- API/Schema/WebSocket：08*API契约与Schema字典.md + 08A*原子API与Schema展开.md + 08B\_原子Schema字段字典.md
- 页面/交互：09*页面与交互规格.md + 09A*关键页面逐页交互展开.md
- AI/Agent/自动化/集成：10*AI_Agent*自动化与集成执行规范.md
- 商业化/交易/计费：11*商业化*交易与计费体系.md
- 测试/验收/部署/迁移：12*测试*验收*部署与迁移.md + 12A*原子TEST_ACPT与发布门禁展开.md
- 历史映射与归档：13\_历史映射与归档说明.md

3. 实现级输入规则

- 如果任务已进入编码/联调/测试阶段，优先使用实现级子文档：
  - 05A
  - 08A
  - 08B
  - 09A
  - 12A
- 主文档负责索引与主规则，子文档负责实现粒度
- 不允许跳过实现级子文档直接凭空生成代码

========================
二、阶段与范围规则
========================

1. 默认阶段

- 除非任务明确指定 S2/S3/S4，否则默认按 S1 开发
- 不要擅自把终局版能力提前实装到当前阶段
- 可以参考终局范围，但实现必须服从当前阶段边界

2. 每次新任务都必须先判定：

- 当前任务属于哪个阶段：S1 / S2 / S3 / S4
- 影响哪些 REQ
- 影响哪些 TABLE / SM / PERM / API / PAGE / TEST / ACPT
- 是否属于当前阶段 required_in 范围
- 是否会触发 must_for_release 的发布门禁变化

3. 任务范围确定流程
   每次新任务必须按以下顺序推进：

- 在 01 中确认阶段边界
- 在 03 中定位 REQ
- 在 04 中提取闭环链路
- 在 05~12 中读取事实定义
- 在 05A/08A/08B/09A/12A 中读取实现级输入
- 再开始编码

========================
三、开发执行规则
========================

1. 严禁臆造事实

- 没有写进 SSOT 的事实，不要直接当成既定实现
- 如果文档缺少关键细节：
  - 先判断是否可由现有事实推导
  - 可推导则采用最保守、最一致的方案，并显式说明
  - 不可推导则输出“文档缺口/冲突”，暂停该点扩展实现

2. 开发必须遵守的核心底线

- 多租户隔离必须基于 org_id
- 路径参数统一使用 {id}
- 首页根路径只允许重定向到 /dashboard
- 商机结果只允许 won / lost
- 会话权限必须细分 SEND / ACCEPT / TRANSFER / CLOSE
- 所有状态变更必须符合状态机，不允许私自增加非法流转
- 所有敏感写操作必须考虑权限、审计、并发版本控制
- 任何 AI 自动执行、高风险动作、审批、回滚、接管，必须遵守 07 和 10
- 任何商业化、计费、账单、支付、发票、续费、停服逻辑，必须遵守 11

3. 默认开发粒度
   每次任务尽量拆成以下可执行层次：

- schema / migration
- repository / service
- API / contract
- page / interaction
- test
- doc sync

4. 变更前必须输出影响分析
   开始编码前，必须先列出：

- 目标 REQ
- 目标模块
- 涉及文档
- 涉及文件
- 涉及表
- 涉及接口
- 涉及页面
- 涉及测试
- 是否需要文档回写

5. 不允许的行为

- 不允许绕开 SSOT 自行重新定义表结构、状态、接口、页面
- 不允许只改代码不补测试
- 不允许只改代码不判断文档是否需要同步
- 不允许把历史文档当当前实现基线
- 不允许在未识别阶段范围前直接大面积编码
- 不允许为了省事跳过权限、审计、幂等、错误处理

========================
四、任务编排规则
========================

每次新建任务时，按以下固定模板执行：

【步骤 1：任务识别】
输出：

- 任务目标
- 当前阶段
- 影响模块
- 目标 REQ-ID
- 是否属于当前阶段发布范围

【步骤 2：文档读取】
至少读取并引用：

- 00
- 01
- 03
- 04
  并补读对应事实文档与实现级子文档

【步骤 3：开发计划】
输出：

- 本次要改的文件
- 每个文件改什么
- 开发顺序
- 依赖关系
- 风险点
- 验证方式

【步骤 4：实施】
按计划逐步执行，优先做最小闭环，不做无边界扩展

【步骤 5：验证】
至少验证：

- 功能正确性
- 权限
- 状态机
- 错误路径
- 测试是否补齐
  必要时增加：
- 审计
- WebSocket
- 并发/版本控制
- 多租户隔离
- 幂等

【步骤 6：文档回写判断】
结束前必须检查：

- 是否有新事实产生
- 是否影响现有事实
- 哪些文档必须同步更新

【步骤 7：总结输出】
输出：

- 实际修改文件
- 完成情况
- 风险残留
- 验证情况
- 文档更新情况
- 下一步建议

========================
五、文档回写规则
========================

1. 回写总原则
   所有事实只定义一次，必须回写到事实归属文档，而不是随便写到其他文档里。

2. 固定回写顺序
   发现事实变化后，按以下顺序回写：

- 先定位事实归属文档
- 更新事实归属文档
- 更新受影响文档
- 更新 04\_需求追踪矩阵.md
- 如果涉及旧口径映射、裁决、迁移说明，再更新 13\_历史映射与归档说明.md

3. 常见回写链路

- 新增需求：03 -> 04 -> 05/06/07/08/09/12 -> 13(如需要)
- 新增表/字段：05 -> 08(如接口变化) -> 09(如页面变化) -> 12 -> 04
- 新增状态流转：06 -> 08 -> 09 -> 12 -> 04
- 新增权限/数据范围：07 -> 08 -> 09 -> 12 -> 04
- 新增 API / Schema / WS：08 -> 09 -> 12 -> 04
- 新增页面：09 -> 08 -> 07 -> 12 -> 04
- 新增 Agent/自动化/集成流程：10 -> 05/06/07/08/09/12 -> 04
- 商业化规则变化：11 -> 05/06/07/08/09/12 -> 04
- 部署/迁移变更：12 -> 05/08(如需要) -> 13(如需要)

4. 代码与文档同步策略

- 如果实现改变了文档中的事实，代码提交与文档提交必须同批完成
- 如果实现还没改事实，只补充实现细节，则优先更新实现级子文档
- 不允许文档长期落后于代码

========================
六、测试与发布门禁规则
========================

1. 每次任务都要映射测试

- 每次开发都要定位对应 TEST-ID / ACPT-ID
- 没有测试映射的改动，不算完整开发

2. 验证等级
   按任务复杂度选择：

- 小改动：至少补齐单元测试或接口测试
- 跨前后端：补接口 + 页面交互验证
- 状态机/审批/AI/商业化/集成：必须补充集成测试或 E2E 验证方案

3. 阻塞条件
   出现以下情况时，不允许把任务标记为完成：

- 当前阶段 required_in 范围内的关键链路未闭环
- 状态机未对齐
- 权限未验证
- 审计未考虑
- 文档应改未改
- 测试应补未补

========================
七、输出格式规则
========================

默认按以下结构输出：

1. 任务理解
2. 文档依据
3. 影响范围
4. 开发计划
5. 实施结果
6. 验证结果
7. 文档回写
8. 风险与下一步

要求：

- 简洁但不能缺关键项
- 以可执行为中心
- 明确指出“已完成 / 未完成 / 待确认”
- 不把推测写成事实

========================
八、项目目标对齐规则
========================

你必须始终记住：

- MOY 不是普通工具型产品
- MOY 面向的是企业“从获客-沟通-谈单-成交-售后-续费”的结果交付链路
- 代码实现不能只追求局部功能跑通，而要尽量服从整条业务主链
- 即使当前默认开发 S1，也要避免做出会破坏未来 S2/S3/S4 扩展能力的实现
- 但也不能为了终局抽象而过度设计，当前任务仍然优先服从本阶段边界
