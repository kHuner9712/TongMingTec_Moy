# MOY 根目录文档阅读整理（基于 V0.1）

## 1. 文档来源与定位
- 来源文件：`MOY_文档体系总纲_与_Codex生成协议_V0.1.docx`
- 文档性质：项目文档体系的“总控蓝图”，不是 26 份文档正文。
- 目标：统一 MOY 从立项到运维全流程中的人、流程、AI 代理协作边界，降低返工和方向漂移。

## 2. 项目一页式定义（提炼）
### 2.1 项目定义
MOY（墨言）是桐鸣科技打造的企业级 AI 原生客户经营系统，面向销售与服务一体化场景，聚焦结果交付（转化、效率、生命周期价值）。

### 2.2 核心链路
获客 -> 沟通 -> 谈单 -> 成交 -> 售后 -> 复购

### 2.3 价值主张
不是“会聊天的 AI 工具”，而是“可交付经营结果的系统”。

### 2.4 首期范围（MVP）
- 终端策略：Web 端优先。
- 建设策略：架构按全链路设计，但交付范围先收敛为可闭环、可验证、可收费。

### 2.5 北极星指标
- 线索转化率
- 首响时效
- 跟进及时率
- 成交周期
- 客户满意度
- 复购率
- 单客户生命周期价值

## 3. 文档治理原则（执行基线）
1. 单一事实源：同一业务定义只在一个主文档拥有最终解释权。
2. 先文档后开发：核心链路、边界、验收未定前，不进入大规模编码。
3. 版本化管理：所有文档必须记录版本、变更人、日期、摘要、审批状态。
4. 可追溯性：业务需求需可追溯到 PRD、设计、开发、测试、上线验证。
5. 显式假设：未知信息必须标注【待确认】/【假设】/TODO。
6. 阶段化产出：按阶段迭代，不追求一次性完成 26 份文档。

## 5. 依赖关系与推荐顺序
### 5.1 推荐顺序（最小可行链）
项目介绍 -> 可行性研究 -> 项目章程/计划 -> BRD -> PRD -> 需求跟踪矩阵 -> HLD -> DBD -> API -> 开发规范

### 5.2 关键依赖
- BRD 依赖：项目介绍、可行性研究、调研记录
- PRD 依赖：BRD、章程、对象模型、范围边界
- HLD 依赖：PRD 功能范围、非功能、接口边界
- LLD 依赖：HLD、API、DBD、专项设计
- 测试用例依赖：PRD、RTM、API、LLD、验收标准
- 上线验收依赖：测试总结、遗留缺陷评估、回滚与监控方案

## 7. Codex/Trae 协作协议（可直接执行）
1. 生成任何文档前必须先读取：总纲 + 最新项目介绍 + BRD + PRD + RTM。
2. 默认中文输出；主文档统一中文。
3. 未确认信息必须显式标注【待确认】或【假设】。
4. 输出骨架固定为：目的 -> 范围 -> 输入 -> 正文 -> 验收 -> 待确认项。
5. 涉及安全、隐私、合同、合规、财务估算必须写明假设前提。
6. 任何文档改动都要写变更摘要，并同步更新 RTM。
7. AI 角色定位：文档工程师，不得擅自改变产品方向。

## 8. 防飘移机制（治理重点）
- 每次生成新文档必须指定上游输入，缺失时允许输出草稿但不得伪造完整结论。
- 先草稿评审后入库，重点审查战略一致性、范围边界、指标可验证性。
- 所有数值结论必须附计算口径与保守/中性/乐观假设。
- 合规内容仅做风险识别，不给法律绝对结论。
- PRD 变更必须反向同步 BRD、RTM、HLD、API、测试用例。

## 9. 立即可执行的启动动作（原文建议）
1. 先生成《MOY 项目介绍》第一版。
2. 再生成《MOY 可行性研究报告》。
3. 沉淀《MOY BRD》并稳定业务链路与指标。
4. BRD 稳定后进入 PRD、HLD、DBD、API。
5. 从第一天开始维护 RTM 和版本记录。

## 10. 本次阅读结论

- 该文档已提供完整的"项目文档操作系统"：定义了文档清单、优先级、依赖链、命名规则、AI 协作协议与防飘移机制。
- 当前最关键不是补写全部文档，而是按 P0 顺序逐份生成、评审、入库，确保"同一事实源 + 可追溯 + 可验收"。
- 若按该基线执行，可显著降低后续开发返工、跨角色理解偏差与 AI 输出失真风险。
