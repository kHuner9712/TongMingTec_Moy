# MOY 对象模型与数据库设计

## 1. 文档定位
本文档是 MOY 唯一 DBD / 数据主文档，负责：
- 对象说明
- 表结构与字段
- 关系与级联
- 索引、审计、软删除
- 多租户 `org_id` 策略
- 初始化数据
- 迁移顺序

### 1.1 实现级展开入口
- 本文档继续承担对象、表清单、统一建模规则与迁移总顺序的唯一主链职责。
- 终局关键表的 migration-ready / DDL-ready 输入，统一以下列实现级子文档为准：
  - [05A_终局关键表DDL展开.md](./05A_终局关键表DDL展开.md)
- `05` 中的表清单用于索引与裁决；字段级可空性、默认值、唯一约束、部分索引、级联策略，以 `05A` 为最终实现输入。

## 2. 建模总规范
### 2.1 数据库与命名
- 数据库：PostgreSQL 15+
- 表名：`snake_case` 复数
- 主键：默认 `uuid`
- 审计日志主键：`bigserial`
- 时间：`timestamptz`
- JSON：`jsonb`
- 货币：`numeric(14,2)`
- AI 成本/倍率：`numeric(12,6)`

### 2.2 多租户隔离
- 除全局部署/许可表外，业务表必须带 `org_id uuid not null`。
- Repository、Service、SQL View 默认先过滤 `org_id`。
- 跨租户访问统一拒绝，不允许通过前端绕过。

### 2.3 路径与并发约束
- 所有可变主业务表都带 `version int not null default 1`。
- 更新类 API 必须携带 `version`。
- 与状态流转有关的表必须保留状态历史或事件日志。

### 2.4 字段模板
#### PROFILE-MUTABLE
适用于主业务表：
- `id uuid primary key not null`
- `org_id uuid not null`：默认建立复合索引首列；引用 `organizations.id`，`on delete restrict / on update cascade`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`
- `created_by uuid null`：默认引用 `users.id`，`on delete set null / on update cascade`
- `updated_by uuid null`：默认引用 `users.id`，`on delete set null / on update cascade`
- `deleted_at timestamptz null default null`
- `version int not null default 1 check(version >= 1)`
- 继承规则：
  - 所有 `PROFILE-MUTABLE` 表默认启用软删除语义。
  - 默认建立 `idx_{table}_org_status_time(org_id,status,updated_at desc)`；无 `status` 字段的表则退化为 `idx_{table}_org_updated(org_id,updated_at desc)`。
  - 所有唯一键默认写成 `... WHERE deleted_at IS NULL` 的 partial unique index，除非该表被 `05A` 明确声明为禁止软删除。

#### PROFILE-APPEND
适用于追加日志表：
- `id uuid primary key not null`
- `org_id uuid not null`：默认建立 `idx_{table}_org_created(org_id,created_at desc)`；引用 `organizations.id`，`on delete restrict / on update cascade`
- `created_at timestamptz not null default now()`
- `created_by uuid null`：默认引用 `users.id`，`on delete set null / on update cascade`
- 继承规则：
  - `PROFILE-APPEND` 表默认禁止更新与物理删除，只允许追加。
  - 若需纠错，必须通过补偿记录，不直接修改历史行。

#### PROFILE-SYSTEM
适用于部署/许可/投递类表：
- `id uuid primary key not null`
- `org_id uuid not null`：若为租户级系统表，默认建立 `idx_{table}_org_updated(org_id,updated_at desc)`；若 `05A` 明确为跨租户部署控制表，则仍保留 `org_id` 作为授权归属字段
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`
- `version int not null default 1 check(version >= 1)`
- 继承规则：
  - `PROFILE-SYSTEM` 表默认禁止软删除，除非 `05A` 明确声明支持软删除。
  - 涉及部署、许可、迁移批次的表，默认要求保留审计与回滚定位字段。

## 5. 核心表定义
以下只定义业务字段；共享字段继承对应 `PROFILE`。

### 5.1 平台基础
#### TABLE-ORG-001 organizations
- 说明：组织主体，是 `org_id` 的来源。
- 业务字段：
  - `code varchar(64)`：全局唯一组织编码
  - `name varchar(128)`：组织名称
  - `status varchar(16)`：`active/disabled`
  - `timezone varchar(64)`：默认时区
  - `locale varchar(16)`：默认区域
  - `logo_url varchar(255)`：Logo
  - `owner_user_id uuid null`
- 键/索引：`uq(code)`、`idx(status)`。
- 级联：被 `users/departments/...` 外键引用，禁止硬删除。
- 初始化：创建租户时插入一条。

#### TABLE-ORG-002 departments
- 业务字段：
  - `parent_id uuid null`
  - `code varchar(64)`
  - `name varchar(128)`
  - `manager_user_id uuid null`
  - `sort_order int`
  - `is_active boolean`
- 键/索引：`uq(org_id,code)`、`idx(org_id,parent_id)`。
- 级联：软删部门前必须迁移子部门/成员。

#### TABLE-USR-001 users
- 业务字段：
  - `department_id uuid null`
  - `username varchar(64)`
  - `display_name varchar(64)`
  - `email varchar(128) null`
  - `mobile varchar(32) null`
  - `password_hash varchar(255)`
  - `status varchar(16)`：`active/inactive/locked`
  - `last_login_at timestamptz null`
- 键/索引：`uq(org_id,username)`、`uq(org_id,email)`、`idx(org_id,department_id)`。
- 级联：角色关系表删除时联删关系，不联删用户。

#### TABLE-USR-002 roles
- 业务字段：
  - `code varchar(64)`
  - `name varchar(64)`
  - `data_scope varchar(16)`：`self/team/org`
  - `is_default boolean`
- 键/索引：`uq(org_id,code)`。

#### TABLE-USR-003 permissions
- 业务字段：
  - `perm_id varchar(64)`：格式 `PERM-*`
  - `module varchar(32)`
  - `action varchar(64)`
  - `description varchar(255) null`
- 键/索引：`uq(org_id,perm_id)`、`idx(org_id,module)`。
- 初始化：租户开通时写入 S1 必需权限，S2+ 由阶段升级脚本补齐。

#### TABLE-USR-004 user_roles
- 业务字段：
  - `user_id uuid`
  - `role_id uuid`
  - `source varchar(16)`：`manual/bootstrap/sync`
  - `effective_from timestamptz null`
  - `effective_to timestamptz null`
- 键/索引：`uq(org_id,user_id,role_id)`、`idx(org_id,role_id)`。
- 级联：删除角色或用户时仅删除关系，不联删主数据。

#### TABLE-USR-005 role_permissions
- 业务字段：
  - `role_id uuid`
  - `permission_id uuid`
  - `scope_override jsonb null`
  - `granted_by uuid null`
- 键/索引：`uq(org_id,role_id,permission_id)`、`idx(org_id,permission_id)`。
- 初始化：默认角色权限由租户初始化脚本写入。

#### TABLE-USR-006 field_scope_rules
- 业务字段：
  - `resource_type varchar(64)`
  - `field_name varchar(64)`
  - `scope_mode varchar(16)`：`full/masked/hidden`
  - `rule_expr jsonb`
  - `priority int`
- 键/索引：`idx(org_id,resource_type)`。

#### TABLE-USR-007 permission_change_logs
- 业务字段：
  - `role_id uuid`
  - `change_type varchar(32)`：`grant/revoke/rollback`
  - `before_snapshot jsonb`
  - `after_snapshot jsonb`
  - `request_id uuid null`

### 5.2 客户、线索、商机
#### TABLE-CM-001 customers
- 业务字段：
  - `name varchar(128)`
  - `industry varchar(64) null`
  - `level varchar(16) null`
  - `owner_user_id uuid`
  - `status varchar(16)`：`potential/active/silent/lost`
  - `phone varchar(32) null`
  - `email varchar(128) null`
  - `address varchar(255) null`
  - `remark text null`
- 索引：`idx(org_id,status)`、`idx(org_id,owner_user_id)`、`idx(org_id,name gin_trgm_ops)`。

#### TABLE-CM-002 customer_contacts
- 业务字段：
  - `customer_id uuid`
  - `name varchar(64)`
  - `title varchar(64) null`
  - `phone varchar(32) null`
  - `email varchar(128) null`
  - `wechat varchar(64) null`

## 6. 索引与约束策略
- 所有主业务表默认索引：`idx_{table}_org_status`、`idx_{table}_org_updated_at`。
- 所有外键列必须建索引。
- 所有业务唯一编码采用租户内唯一：`uq(org_id, code)` 或 `uq(org_id, xxx_no)`。
- 软删除表的唯一约束建议使用部分唯一索引：`where deleted_at is null`。

## 7. 初始化数据
必须初始化：
- `organizations`
- 根 `departments`
- 默认 `roles`
- 默认 `permissions`
- 默认 `system_configs`
- S1 基础 `channels`
- S3 以后默认 `plans`

## 8. 级联规则
- 主业务表默认软删除，不做数据库层硬联删。
- 关系表、日志表可物理保留，不随主表硬删。
- 合同、订单、账单、支付、发票一旦进入审计或财务状态，禁止物理删除。
