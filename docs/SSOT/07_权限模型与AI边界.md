# MOY 权限模型与 AI 边界

## 1. 文档定位
本文档是 MOY 唯一权限基线，统一定义：
- RBAC
- data scope / row scope / field scope
- 人工角色权限
- AI Agent 权限边界
- 敏感操作分级
- 审批要求
- 审计要求
- 多租户隔离

## 2. 基础模型
- 权限模型：`RBAC + data scope + row scope + field scope`
- 权限编码：`PERM-{MOD}-{ACTION}`
- 默认隔离：所有资源先按 `org_id` 过滤
- P0 强约束继续保留：
  - 会话权限必须拆分 `PERM-CNV-SEND / ACCEPT / TRANSFER / CLOSE`
  - 不允许定义聚合权限 `conversation:handle`

## 4. data scope / row scope / field scope
### 4.1 data scope
| 值 | 说明 |
| --- | --- |
| `self` | 本人负责、本人创建或本人被指派的数据 |
| `team` | 本人 + 下属团队数据 |
| `org` | 当前租户内全部数据 |

### 4.2 row scope
行级访问规则由资源类型决定：
- `customers/leads/opportunities/tickets/tasks/conversations`：按 `owner_user_id / assignee_user_id / creator_id` 计算。
- `roles/permissions/system_configs/channels/plans/bills`：默认 `org` 级。
- `portal` 侧资源：按客户主体、账单主体或联系人主体计算。

### 4.3 field scope
| 字段级别 | 示例字段 | 默认规则 |
| --- | --- | --- |
| `public` | `name`, `status`, `created_at` | 有行权限即可读 |
| `business_sensitive` | `amount`, `score_reason`, `solution` | 需模块查看权限 |
| `personal_sensitive` | `mobile`, `email`, `address`, `tax_no` | 允许 `masked/full/hidden` |
| `financial_sensitive` | `payment_no`, `invoice_no`, `tax_amount` | 默认仅财务/管理员可全量读 |
| `ai_sensitive` | `prompt`, `tool_scope`, `output_payload` | 仅 AI 管理员与审计角色可全量读 |

`TABLE-USR-006 field_scope_rules` 负责落地字段权限，优先级：
1. 明确 deny
2. field_scope_rule
3. role data_scope
4. 模块默认规则

## 7. AI Agent 权限边界
### 7.1 AI 角色
| AI 角色 | 说明 | 默认模式 |
| --- | --- | --- |
| AI Assistant | 只生成建议，不直接落关键终态 | `suggest` |
| AI Operator | 可执行低风险自动动作 | `assist/auto` |
| AI Approver | 处理 AI 审批请求 | `approval` |
| AI Auditor | 查看 AI 执行、回滚、接管与质检日志 | `suggest` |

### 7.2 AI 可执行操作
- 读取本租户授权资源的非敏感字段
- 生成回复、摘要、分类、评分建议
- 创建 AI task、Agent run、质检报告、自动化执行记录
- 在显式授权下触发低风险动作：
  - 草稿保存
  - 标签建议
  - 指标分析
  - 提醒创建

### 7.3 AI 禁止操作
- 跨 `org_id` 读取或写入
- 绕过审批直接修改财务终态
- 直接关闭工单、直接赢单、直接发票开具、直接支付成功落账
- 修改角色权限、关闭审计、删除账单、清空日志

### 7.4 需要审批的 AI 操作
| 操作 | 风险等级 | 审批要求 |
| --- | --- | --- |
| 自动转派会话/工单到跨团队人员 | P2 | 经理审批 |
| 批量修改客户/线索/商机状态 | P2 | 模块经理审批 |
| 创建或启用 `auto` 模式 Agent | P1 | AI_OPERATOR + ADMIN 双审批 |
| 支付/发票/欠费/停服相关动作 | P0 | 财务审批，不允许 AI 直接提交终态 |
| 权限回滚、字段权限变更 | P0 | ADMIN + AUDIT |

### 7.5 AI 可操作模块边界
| 模块 | 允许模式 | 边界说明 |
| --- | --- | --- |
| QT | suggest/assist/auto | AI 可根据商机信息自动生成报价草稿，审批模式需人工确认 |
| CT | suggest/assist/auto | AI 可根据报价自动生成合同草稿，签署必须人工确认 |
| ORD | suggest/assist | AI 可建议从合同生成订单，订单确认必须人工操作 |
| PAY | suggest | AI 可建议付款提醒，付款确认必须人工操作 |

### 7.6 高风险动作清单
| 动作 | 模块 | 审批模式 | 说明 |
| --- | --- | --- | --- |
| 合同签署 | CT | approval | 合同一经签署即产生法律效力 |
| 订单确认 | ORD | approval | 订单确认后触发付款和开通链路 |
| 付款确认 | PAY | approval | 付款确认涉及资金流，必须人工审批 |

## 8. 敏感操作分级
| 分级 | 说明 | 示例 |
| --- | --- | --- |
| P0 | 极高风险，必须人工最终确认 | 发票开具、支付对账、权限回滚、停服 |
| P1 | 高风险，需要审批 | 启用 auto Agent、合同审批通过、批量状态变更 |
| P2 | 中风险，可在策略下审批或双确认 | 自动转派、营销批量触达、Webhook 激活 |
| P3 | 低风险，可直接执行并审计 | 生成建议、创建提醒、查询报表 |

## 9. 审计要求
- 所有权限拒绝、审批结果、AI 接管、AI 回滚、字段脱敏命中都要落审计。
- 审计最少字段：
  - `request_id`
  - `org_id`
  - `operator_type`
  - `operator_id`
  - `resource_type`
  - `resource_id`
  - `action`
  - `result`
  - `error_code`

## 10. 实现约束
- 后端强校验，前端只做显隐与友好提示。
- 所有页面按钮显隐必须与 `08` API 权限映射一致。
- 所有 AI 工具调用必须先经过权限解析层，而不是在 Agent 内自行判断。
