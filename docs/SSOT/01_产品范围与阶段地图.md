# MOY 产品范围与阶段地图

## 1. 文档目标
本文档只回答两件事：
- 终局版 MOY 的范围上限是什么。
- 同一套终局能力如何按 `S1/S2/S3/S4` 落地，而不是复制出多套文档。

## 2. 终局版产品边界
MOY 的终局边界是一个以 `org_id` 为核心隔离单元的 AI 原生客户经营系统，覆盖：
- 获客：线索、渠道、分群、营销自动化、导入去重、评分与回收
- 沟通：统一会话、消息、AI 辅助、实时协作、客户自助与移动触点
- 商机：商机推进、报价、合同、订单、回款/支付、赢输单与续费触发
- 服务：工单、SLA、服务记录、知识库、客户门户、质检与审计
- 客户成功：健康度、成功计划、续费、增购、流失预警
- AI：智能回复、Agent registry、审批、回滚、接管、质量分析、自动化编排
- 商业化：plan、addon、quota、subscription、bill、payment、invoice、overdue、suspension
- 平台：开放 API、Webhook、SDK、开发者中心、集成流、数据映射
- 国际化与部署：多语言、多区域、GDPR/数据本地化、私有化、迁移与多环境运维

## 5. 阶段落地路径
### 5.1 S1
- 范围：`AUTH ORG USR CM LM OM CNV TK TSK NTF CHN AI AUD SYS COR CMEM ART APC TKC RBC`
- 目标：以 AI 运行时为内核，把 `客户经营 -> 线索 -> 商机`、`会话 -> 工单`、`AI 执行流 -> 审批/接管/回滚` 三条主链做成实现级闭环；前端以经营驾驶舱和工作台为主入口。
- 硬约束继承 P0：
  - 路径参数统一 `{id}`
  - `/` 重定向到 `/cockpit`（经营驾驶舱）
  - 商机结果只允许 `won/lost`
  - 会话权限必须细分 `SEND/ACCEPT/TRANSFER/CLOSE`
  - 多租户隔离必须基于 `org_id`
  - AI 执行必须通过 Agent Runtime，不允许绕开审批/接管/回滚机制

### 5.2 S2
- 范围：在 S1 上新增 `KB DASH AUTO QT CT CSM` 与 AI/权限增强。
- 目标：补齐运营闭环、成交前链路、知识复用与客户成功基础能力。
- 重点：知识库、业务看板、营销自动化、报价、合同、健康度、续费提醒、Agent 审批。

### 5.3 S3
- 范围：在 S2 上新增 `ORD PLAN SUB BILL PAY INV INT PLT` 与扩展渠道。
- 目标：把“谈单 -> 成交 -> 订阅 -> 账单 -> 回款 -> 发票 -> 续费”做成商业闭环，并开放集成能力。
- 重点：订单、订阅、账单、支付、发票、开放 API、Webhook、移动端、多渠道。

### 5.4 S4
- 范围：在 S3 上新增 `I18N DEPLOY` 及终局规模化 Agent / 生态 / 合规能力。
- 目标：支撑全球化、多部署形态、复杂组织规模和高度自动化运营。
- 重点：国际化、多区域、私有化、数据本地化、开发者生态、规模化 AI 编排与审计。

## 7. 终局范围上限
以下能力属于终局版 MOY 上限，必须纳入 SSOT 范围，但不要求在 S1 一次性交付：
- 全渠道获客与统一路由
- 全链路成交与订阅计费
- 客户成功与续费增购
- Agent registry、approval、rollback、takeover、quality、insight、orchestrate
- Webhook、开放 API、SDK、开发者中心、应用生态
- 国际化、多区域、私有化、离线激活、迁移工具

## 8. 不在本项目终局版范围内
- 自研 ERP / 财务总账
- 自研呼叫中心底层交换机
- 自研电子签平台
- 自研支付清结算通道
- 自研 MDM / 数据中台平台产品

上述能力如需支持，统一通过 `INT` 或合作方接入，不在 SSOT 中拆成独立产品主干。
