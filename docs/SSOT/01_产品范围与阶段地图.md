# MOY 产品范围与阶段地图

## 1. 文档目标

本文档只回答两件事：

- 终局版 MOY 的范围上限是什么。
- 同一套终局能力如何按 `S1/S2/S3/S4` 落地，而不是复制出多套文档。

## 2. 终局版产品边界

MOY 的终局边界是一个以 `org_id` 为核心隔离单元的 AI 原生客户经营系统，覆盖：

- 获客：线索、渠道、分群、营销自动化、导入去重、评分与回收
- 沟通：统一会话、消息、AI 辅助、实时协作、客户自助与移动触点
- 商机：商机推进、报价、合同、订单、回款/支付、赢输单与续费触发
- 交付：交付单/实施单、里程碑、交付任务、验收记录、风险项、客户承诺结果
- 服务：工单、SLA、服务记录、知识库、客户门户、质检与审计
- 客户成功：健康度、成功计划、续费、增购、流失预警
- AI：智能回复、Agent registry、审批、回滚、接管、质量分析、自动化编排
- 商业化：plan、addon、quota、subscription、bill、payment、invoice、overdue、suspension
- 平台：开放 API、Webhook、SDK、开发者中心、集成流、数据映射
- 国际化与部署：多语言、多区域、GDPR/数据本地化、私有化、迁移与多环境运维

## 3. 阶段定义

| 阶段 | 名称                 | 定义                                                                                                                                                    | 交付目标                         |
| ---- | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------- |
| S1   | 基础底座与前链路主干 | 建立多租户基础底座、前链路经营（客户/线索/商机/会话/工单）、AI runtime 与治理能力（审批/接管/回滚）                                                     | 前链路经营闭环 + AI 受控落地     |

| S3   | 商业化与平台化扩展   | 在 S2 成交后衔接主干基础上，补齐完整商业化闭环（套餐/账单/发票/对账/欠费/续费）与平台化能力（开放 API/Webhook/SDK/移动端/扩展渠道）                     | 从成交闭环走向商业化运营与平台化 |

### 3.1 模块成熟度分级定义

| 级别               | 定义                        | 判定标准                                                               |
| ------------------ | --------------------------- | ---------------------------------------------------------------------- |
| `skeleton`         | 只有骨架/占位，不能视为完成 | 路由存在但页面无实质内容；或 Entity 存在但无完整 CRUD API              |
| `crud_ready`       | 基础增删改查可用            | 列表/详情/创建/编辑 API 可调通，基础数据可持久化，但关键业务流程未跑通 |
| `workflow_ready`   | 关键业务流程可跑通          | 状态机流转正确，跨模块关联可走通，审批/通知等流程节点可用              |
| `production_ready` | 具备上线级可靠性            | 权限/审计/异常处理/并发版本控制/观测能力齐备，测试覆盖达标             |

模块当前成熟度标注见 02\_业务域与模块树.md。

## 5. 阶段落地路径

### 5.1 S1

- 当前状态：**基本完成**
- 范围：`AUTH ORG USR CM LM OM CNV TK TSK NTF CHN AI AUD SYS COR CMEM ART APC TKC RBC`
- 目标：以 AI 运行时为内核，把 `客户经营 -> 线索 -> 商机`、`会话 -> 工单`、`AI 执行流 -> 审批/接管/回滚` 三条主链做成实现级闭环；前端以经营驾驶舱和工作台为主入口。
- 硬约束继承 P0：
  - 路径参数统一 `{id}`
  - `/` 重定向到 `/cockpit`（经营驾驶舱）
  - 商机结果只允许 `won/lost`
  - 会话权限必须细分 `SEND/ACCEPT/TRANSFER/CLOSE`
  - 多租户隔离必须基于 `org_id`
  - AI 执行必须通过 Agent Runtime，不允许绕开审批/接管/回滚机制

#### DLV 转正说明（结果交付型系统定位）

- 为什么存在：DLV 是“订单/订阅开通”到“结果达成可验收”的承接层，避免系统停留在“成交即结束”的伪闭环。
- introduced_in / required_in：`introduced_in=S2`，`required_in=S3`（S2 先落地最小可用主干，S3 补齐生产级治理与规模化运营能力）。
- 当前成熟度：`workflow_ready`（状态机、关键 API、事件驱动衔接、前端最小入口均已落地）。
- 与 S2/S3 关系：S2 负责跑通“交付实施→验收”与 CSM 读数联动；S3 在此基础上补齐更完整的商业化与治理闭环。

## 10. 终局范围上限

以下能力属于终局版 MOY 上限，必须纳入 SSOT 范围，但不要求在 S1 一次性交付：

- 全渠道获客与统一路由
- 全链路成交与订阅计费
- 客户成功与续费增购
- Agent registry、approval、rollback、takeover、quality、insight、orchestrate
- Webhook、开放 API、SDK、开发者中心、应用生态
- 国际化、多区域、私有化、离线激活、迁移工具

## 11. 不在本项目终局版范围内

- 自研 ERP / 财务总账
- 自研呼叫中心底层交换机
- 自研电子签平台
- 自研支付清结算通道
- 自研 MDM / 数据中台平台产品

上述能力如需支持，统一通过 `INT` 或合作方接入，不在 SSOT 中拆成独立产品主干。
