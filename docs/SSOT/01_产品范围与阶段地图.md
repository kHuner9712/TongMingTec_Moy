# MOY 产品范围与阶段地图

## 1. 文档目标
本文档只回答两件事：
- 终局版 MOY 的范围上限是什么。
- 同一套终局能力如何按 `S1/S2/S3/S4` 落地，而不是复制出多套文档。

## 2. 终局版产品边界
MOY 的终局边界是一个以 `org_id` 为核心隔离单元的 AI 原生客户经营系统，覆盖：
- 获客：线索、渠道、分群、营销自动化、导入去重、评分与回收
- 沟通：统一会话、消息、AI 辅助、实时协作、客户自助与移动触点
- 商机：商机推进、报价、合同、订单、回款/支付、赢输单与续费触发
- 服务：工单、SLA、服务记录、知识库、客户门户、质检与审计
- 客户成功：健康度、成功计划、续费、增购、流失预警
- AI：智能回复、Agent registry、审批、回滚、接管、质量分析、自动化编排
- 商业化：plan、addon、quota、subscription、bill、payment、invoice、overdue、suspension
- 平台：开放 API、Webhook、SDK、开发者中心、集成流、数据映射
- 国际化与部署：多语言、多区域、GDPR/数据本地化、私有化、迁移与多环境运维

## 3. 阶段定义
| 阶段 | 名称 | 定义 | 交付目标 |
| --- | --- | --- | --- |
| S1 | 基础底座与前链路主干 | 建立多租户基础底座、前链路经营（客户/线索/商机/会话/工单）、AI runtime 与治理能力（审批/接管/回滚） | 前链路经营闭环 + AI 受控落地 |

### 3.1 模块成熟度分级定义

| 级别 | 定义 | 判定标准 |
| --- | --- | --- |
| `skeleton` | 只有骨架/占位，不能视为完成 | 路由存在但页面无实质内容；或 Entity 存在但无完整 CRUD API |
| `crud_ready` | 基础增删改查可用 | 列表/详情/创建/编辑 API 可调通，基础数据可持久化，但关键业务流程未跑通 |
| `workflow_ready` | 关键业务流程可跑通 | 状态机流转正确，跨模块关联可走通，审批/通知等流程节点可用 |
| `production_ready` | 具备上线级可靠性 | 权限/审计/异常处理/并发版本控制/观测能力齐备，测试覆盖达标 |

模块当前成熟度标注见 02_业务域与模块树.md。

## 5. 阶段落地路径
### 5.1 S1
- 当前状态：**基本完成**
- 范围：`AUTH ORG USR CM LM OM CNV TK TSK NTF CHN AI AUD SYS COR CMEM ART APC TKC RBC`
- 目标：以 AI 运行时为内核，把 `客户经营 -> 线索 -> 商机`、`会话 -> 工单`、`AI 执行流 -> 审批/接管/回滚` 三条主链做成实现级闭环；前端以经营驾驶舱和工作台为主入口。
- 硬约束继承 P0：
  - 路径参数统一 `{id}`
  - `/` 重定向到 `/cockpit`（经营驾驶舱）
  - 商机结果只允许 `won/lost`
  - 会话权限必须细分 `SEND/ACCEPT/TRANSFER/CLOSE`
  - 多租户隔离必须基于 `org_id`
  - AI 执行必须通过 Agent Runtime，不允许绕开审批/接管/回滚机制

**S1 阶段门禁**：

| 门禁维度 | 要求 |
| --- | --- |
| 阶段目标 | 建立多租户基础底座、前链路经营闭环（客户→线索→商机→会话→工单）、AI 运行时与治理能力 |
| 必须完成能力 | AUTH/ORG/USR/SYS/AUD：多租户基础底座；CM/LM/OM：前链路经营闭环；CNV/TK/TSK/NTF/CHN：沟通与服务闭环；AI/COR/CMEM/ART/APC/TKC/RBC：AI 运行时与治理 |
| 核心端到端链路 | 1. 线索录入→分配→跟进→转化客户+商机（LM→CM+OM）2. 会话接入→回复→转工单→关闭会话（CNV→TK）3. 工单创建→分派→处理→解决→关闭（TK 全状态）4. AI 智能回复→人工确认→发送（AI suggest 模式） |
| 模块成熟度要求 | AUTH/ORG/USR/SYS/AUD：production_ready；CM/LM/OM/CNV/TK/TSK/NTF/CHN：workflow_ready；AI/COR/CMEM：workflow_ready；ART/APC/TKC/RBC：crud_ready |

| 指标/数据要求 | 客户/线索/商机/会话/工单核心数据可持久化；审计日志可查询；驾驶舱可展示基础摘要 |
| 测试与验收要求 | S1 must_for_release=Y 的 REQ 全部通过 ACPT；4 条核心端到端链路全部可 UI 操作完成；状态机非法迁移全部被拒绝；多租户隔离验证通过 |
| 非本阶段事项 | 报价/合同/订单/付款/订阅（S2）；知识库/经营驾驶舱增强/自动化（S2）；完整商业化闭环（S3）；国际化/私有化（S4） |

### 5.2 S2
- 当前状态：**已有雏形，未达到阶段完成**
- 范围：在 S1 上新增 `KB DASH AUTO QT CT ORD PAY SUB CSM` 与 AI/权限增强。
- 目标：补齐成交与成交后衔接主干：报价→合同→订单→付款→开通/订阅全链路，以及知识库、经营驾驶舱、触发式自动化、客户成功基础能力。
- 重点：
  - QT 报价管理
  - CT 合同管理
  - ORD 订单最小内核（创建/确认/激活）
  - PAY 付款确认最小内核（记录创建/状态跟踪/确认联动）
  - SUB 服务开通/订阅最小内核（开通/基础状态管理）
  - CSM 客户成功（健康度+续费提醒）
  - KB 知识库最小版（分类/条目/审核/搜索/AI问答）
  - DASH 经营驾驶舱增强（销售看板+服务看板）
  - AUTO 触发式自动化主干（事件驱动/节点执行/失败重试）

### 5.5 当前基线与下一阶段推进重点

**当前基线（S1 基本完成）**：
- 多租户基础底座：AUTH / ORG / USR / SYS
- 前链路经营闭环：CM / LM / OM / CNV / TK / TSK / NTF / CHN
- AI runtime 与治理：AI / COR / CMEM / ART / APC / TKC / RBC
- 审计与系统：AUD

**下一阶段推进重点（S2 已有雏形，未达到阶段完成）**：
- 成交前衔接：QT（报价管理）→ CT（合同管理）
- 成交后衔接主干：ORD 最小内核 → PAY 最小内核 → SUB 最小内核
- 经营增强：CSM（客户成功）+ KB 最小版 + DASH 增强 + AUTO 触发式主干

**全链路主干优先原则**：ORD/PAY/SUB 的最小内核在 S2 落地，确保"商机赢单 → 报价 → 合同 → 订单 → 付款 → 开通/订阅"的成交主链不断裂。完整商业化能力（账单/发票/对账/欠费/续费/席位管理/停服恢复）在 S3 补齐。

## 10. 终局范围上限
以下能力属于终局版 MOY 上限，必须纳入 SSOT 范围，但不要求在 S1 一次性交付：
- 全渠道获客与统一路由
- 全链路成交与订阅计费
- 客户成功与续费增购
- Agent registry、approval、rollback、takeover、quality、insight、orchestrate
- Webhook、开放 API、SDK、开发者中心、应用生态
- 国际化、多区域、私有化、离线激活、迁移工具

## 11. 不在本项目终局版范围内
- 自研 ERP / 财务总账
- 自研呼叫中心底层交换机
- 自研电子签平台
- 自研支付清结算通道
- 自研 MDM / 数据中台平台产品

上述能力如需支持，统一通过 `INT` 或合作方接入，不在 SSOT 中拆成独立产品主干。
