# MOY 商业化、交易与计费体系

## 1. 文档定位
本文档是 MOY 终局版商业规则唯一主文档。

本文档只定义：
- 商业对象边界
- 计费规则
- 套餐/额度/续费/欠费/停服规则
- 成交与续费链路

对象、表、状态机、API、PAGE、TEST 不在本文重复定义，只引用主文档 ID。

### 1.1 最小交易闭环

> **核心立场**：当前下一阶段重点不是完整 billing 平台化，而是成交与成交后衔接主干。

**最小交易闭环**是指从商机赢单到订阅开通的必要对象与动作集合，覆盖 quote→contract→order→payment→subscription 的创建、确认与状态流转，不覆盖 bill/invoice/renewal/overdue/suspension。

当前阶段只冻结最小交易闭环内的必要对象与动作，不等于一次性完成全部商业化深度能力。ORD/PAY/SUB 的最小内核需要前移，以承接合同后的真实业务结果，确保成交链路不断裂。完整商业化能力（账单、发票、对账、退款、续费流程、席位管理、停服恢复）仍在 S3 阶段推进。

## 3. 计费规则
### 3.1 套餐
- `plan` 决定功能边界、seat 限额、默认 quota。
- `addon` 用于扩展功能包或额外额度，不单独形成主订阅。
- `quota` 统一用于消息量、AI 调用量、存储量、API 次数等额度计量。

### 3.2 计费模式
| 模式 | 说明 | 对应对象 |
| --- | --- | --- |
| 按席位 | 按订阅席位计费 | `plan`, `subscription`, `subscription_seats` |
| 按量 | 按 quota 消耗计费 | `quota_account`, `quota_ledger`, `bill_item` |
| 按模块 | 通过 addon 启用额外模块 | `add_on`, `order_item` |
| 按 Agent | 通过 quota 或 addon 管理 Agent 使用额度 | `quota_policy`, `bill_item` |

### 3.3 账期与结算
- 月付、年付为首批标准账期。
- `bill` 为应收主单；`bill_item` 为计费明细；`payment` 负责到账；`invoice` 负责票据。
- 欠费通过 `overdue_event` 驱动催缴、停服与恢复。

## 4. 成交到续费链路

### 4.1 当前阶段：最小交易闭环

> 本阶段只冻结从商机赢单到订阅开通的必要链路，确保成交主干不断裂。

1. `SM-opportunity` 赢单
2. 生成 `quote`
3. 审批并发送 `quote`
4. 报价审批后转 `contract`（REQ-QT-004）
5. 签署 `contract`
6. 合同签署后自动确认成交并生成 `order`（REQ-CT-005）
7. 订单确认后自动创建 `payment` 记录（REQ-ORD-004）
8. 付款确认后订单自动激活（REQ-ORD-004）
9. 订单激活后自动开通 `subscription`（REQ-ORD-004）

### 4.2 终局完整链路

> 在最小交易闭环基础上，S3+ 阶段逐步扩展完整商业化能力。

1. ~~最小交易闭环（4.1）~~
2. 生成 `bill`
3. 申请/开具 `invoice`
4. 到期前触发 `renewal`
5. 逾期则写 `overdue_event`
6. 达到阈值后写 `suspension_record`

## 5. 商业规则

### S2 最小内核商业规则

**订单→付款→开通链路**：
1. 合同签署后可创建订单（手动或自动）
2. 订单确认后可创建付款记录
3. 付款确认后订单自动激活
4. 订单激活后自动开通订阅
5. 订阅状态可在客户 360 视图中查看

**S2 边界**：
- 不支持账单周期和自动出账
- 不支持发票管理
- 不支持对账和冲正
- 不支持退款流程
- 不支持续费流程（仅提醒）
- 不支持席位动态管理（仅记录）
- 不支持停服与恢复

### 5.1 最小交易闭环衔接规则

> 以下规则与 03/04 新增 REQ 对齐，确保成交链路各环节自动衔接不断裂。

| 衔接节点 | 触发条件 | 自动动作 | 对应 REQ |
| --- | --- | --- | --- |
| 报价→合同 | 报价审批通过 + 用户点击"生成合同" | 创建合同草稿，带入报价明细/客户信息/金额条款，建立报价→合同引用 | REQ-QT-004 |
| 合同→商机赢单 | 合同状态变更为"已签署" | 标记关联商机为 won，更新客户经营阶段为"已成交" | REQ-CT-005 |
| 合同→订单 | 合同签署完成 | 从合同自动生成订单草稿，带入合同明细与金额，建立合同→订单引用 | REQ-CT-005 |
| 订单→付款 | 订单状态变更为"已确认" | 自动创建付款记录（状态=待支付），关联订单 | REQ-ORD-004 |
| 付款→订单激活 | 付款记录状态变更为"已确认" | 自动激活关联订单 | REQ-ORD-004 |
| 订单激活→订阅 | 订单状态变更为"已激活" | 自动触发订阅开通流程 | REQ-ORD-004 |

## 7. 欠费与停服策略
| 规则 | 说明 |
| --- | --- |
| 欠费判定 | `bill.status = overdue` 或 `payment` 未在到期前成功 |
| 首次催缴 | 逾期当天 |
| 再次催缴 | `+3d`, `+7d` |
| 停服建议 | 达到停服阈值且订阅仍未恢复 |
| 恢复条件 | 欠费清零或管理员人工恢复 |
