#### 6.2.1 DLV 转正核对（结果交付型系统）

- DLV 为什么存在：承接“成交后开通”到“可验收结果”的交付责任链，避免系统停留在“只成交不交付结果”。
- introduced_in / required_in：`introduced_in=S2`，`required_in=S3`。
- 当前成熟度：`workflow_ready`。
- 与 S2/S3 关系：S2 跑通最小交付主干（状态机 + API + 事件 + 前端入口），S3 补齐更完整的生产级治理与规模化能力。

### 6.6 AUTO 结果链动作闭环补齐（2026-04）

- AUTO 不再仅是“事件触发 + 动作执行引擎”，已补齐“经营异常 -> 自动动作 -> 审批边界 -> 结果回写 -> 人工接管/确认”的闭环最小实现。
- 新增 4 类推荐模板（可安装/更新）：
  - `tpl_contract_expiry_to_csm_followup`：合同到期预警 -> 自动创建 CSM 跟进任务
  - `tpl_dlv_high_risk_to_csm`：DLV 高风险 -> 通知负责人 + 写入 CSM 风险视图（高风险步骤需审批）
  - `tpl_lead_missed_followup_alert`：线索漏跟进率异常 -> 自动提醒负责人
  - `tpl_first_response_attention`：首响时间异常 -> 自动生成服务关注项
- AUTO 运行追踪补齐：
  - run 级：触发事件、触发条件快照、触发者、业务上下文、审批态、失败原因、人工接管/确认信息
  - step 级：审批请求 ID、是否需审批、业务上下文、失败原因、结构化输出（含业务对象引用）
- 事件与模块联动补齐：
  - 新增 `dash.metric_anomaly`（经营异常事件）
  - 扩展 `delivery.risk_reported` 事件上下文（customer/owner/contract/order/subscription）
  - 订阅 `approval.status_changed` 以支持审批通过后恢复执行
- 当前 AUTO 成熟度维持 `workflow_ready`，并形成“结果责任链动作闭环”能力基线；生产级能力（规则冲突治理、重放去重、跨环境运行编排）仍在后续阶段治理。

## 7. 模块设计约束

- 所有业务表默认继承 `org_id` 多租户隔离。
- 所有写接口必须可追溯到对应 `REQ / API / PAGE / TEST`。
- 模块之间的依赖只能通过对象、状态机、API、事件四类契约表达，不允许另立文档体系。
- 商业化与平台化模块不得创建第二套独立命名体系，必须沿用本表模块码。
