### 3.1 原子 TEST / ACPT 展开规则
- `TEST-{MOD}-{NNN}` 与 `ACPT-{MOD}-{NNN}` 是唯一可执行编号；区间写法仅用于矩阵压缩展示。
- 每个原子用例必须绑定：
  - 至少 1 个 `REQ-ID`
  - 至少 1 行 `RTM`
  - 至少 1 个 `API-ID` 或 `PAGE-ID`
  - 涉及写操作时，必须断言 `SM`、`PERM`、`AUDIT`
- 生成与落盘顺序固定为：
  1. 从 `03` 取需求
  2. 从 `04` 拉链路
  3. 从 `05/06/07/08/09` 取断言输入
  4. 在测试工程中生成 unit / integration / contract / e2e / uat 用例

### 3.2 测试文件落盘约定
| 类型 | 建议路径 | 说明 |

### 3.5 S2 成交与成交后衔接主干测试

**QT 报价管理**：
- 功能：报价创建、版本管理、审批流程、发送报价
- 权限：PERM-QT-CREATE/VIEW/APPROVE
- 状态机：SM-quote 全路径覆盖
- 错误路径：重复提交审批、无权限审批、已过期报价操作

**CT 合同管理**：
- 功能：合同创建、审批流程、签署、文档上传
- 权限：PERM-CT-CREATE/VIEW/APPROVE/SIGN
- 状态机：SM-contract 全路径覆盖
- 高风险：合同签署必须审批

**ORD 订单最小内核**：
- 功能：订单创建（从合同/报价生成）、确认、激活、取消
- 权限：PERM-ORD-CREATE/VIEW/CONFIRM/CANCEL
- 状态机：SM-order S2 路径覆盖（draft→confirmed→active→completed, draft→cancelled）
- 联动：订单确认→付款关联→付款确认→订单激活→订阅开通

**PAY 付款确认最小内核**：
- 功能：付款记录创建、确认、列表查询
- 权限：PERM-PAY-CREATE/VIEW/CONFIRM
- 状态机：SM-payment S2 路径覆盖（pending→succeeded）
- 高风险：付款确认必须审批

**SUB 订阅最小内核**：
- 功能：订阅开通、暂停、恢复、取消
- 权限：PERM-SUB-CREATE/VIEW/MANAGE
- 状态机：SM-subscription S2 路径覆盖（trial→active→suspended→active, active→cancelled）

**CSM 客户成功**：
- 功能：健康度评估、成功计划、回访记录、续费提醒
- 权限：PERM-CSM-VIEW/MANAGE

**KB 知识库**：
- 功能：分类管理、条目CRUD、审核流程、搜索、AI问答
- 权限：PERM-KB-MANAGE/VIEW/REVIEW
- 状态机：SM-knowledge_item 全路径覆盖

**DASH 经营驾驶舱增强**：
- 功能：销售看板、服务看板、指标快照
- 权限：PERM-DASH-VIEW

**AUTO 触发式自动化**：
- 功能：流程创建、启用/暂停、执行记录、失败重试
- 权限：PERM-AUTO-MANAGE/VIEW
- 状态机：SM-automation_flow 全路径覆盖

**成交链路 E2E 验收**：
- 场景：商机赢单 → 创建报价 → 报价审批 → 发送报价 → 创建合同 → 合同审批 → 合同签署 → 创建订单 → 订单确认 → 创建付款 → 付款确认 → 订单激活 → 开通订阅
- 验证点：每步状态变更正确、关联关系正确、权限校验通过、审计日志完整

## 4. 核心 E2E 链路
| E2E | 覆盖条目 | 通过标准 |
| --- | --- | --- |
| Lead to Opportunity | LM + OM | 线索可转商机，状态一致，审计完整 |

| Quote to Contract to Order | QT + CT + ORD | 审批、签署、订单激活无断点 |
| Order to Subscription to Bill to Payment | ORD + SUB + BILL + PAY | 支付到账后订阅与账单同步正确 |
| Bill to Invoice | BILL + INV | 账单开票链路完整，下载可用 |

| Auto Flow to Notification | AUTO + NTF + CHN | 自动化执行、通知和审计一致 |
| CSM Renewal Loop | CSM + SUB + BILL | 续费提醒、确认、应用状态正确 |

## 8. 告警
| 告警项 | 阈值 |
| --- | --- |
| API 5xx | 5 分钟 > 2% |
| WebSocket 断连 | 5 分钟 > 5% |
| AI 超时 | 10 分钟 > 10% |
| 审批积压 | `pending > 50` |
| Webhook 连续失败 | 单 webhook 连续 3 次失败 |
| 账单生成失败 | 单批次失败率 > 5% |
| 迁移回滚触发 | 任一迁移批次 `rolled_back = true` |

## 9. 回滚策略
### 9.1 应用回滚
- 蓝绿或滚动发布失败时回退到上一稳定版本。
- API Schema 不兼容变更必须带开关，不允许直接覆盖。

### 9.2 数据回滚
- 所有 migration 必须可回滚。
- 财务和审计表只允许补偿，不允许物理回滚删除。
- AI 自动写入通过 `ai_rollbacks` 做业务回滚。

### 11.2 迁移原则
- 先加表/列，再切流量，再清旧逻辑。
- 所有迁移批次必须写 `TABLE-DEPLOY-002 migration_batches`。
- 历史 `P0` 与 `FINAL` 文档不作为迁移输入，统一以 `docs/SSOT` 为准。

- `db connection`

## 13. 国际化 / 多区域约束
- 文本必须外置到 `locale_resources`
- 金额展示必须受 `region_policy.currency` 控制
- 时间展示必须受 `organization.timezone` 与 `region_policy.timezone` 控制
- 数据本地化部署必须限制跨区复制

## 15. 发布门禁
满足以下条件才可发布：

**通用条件（所有阶段）**：
- 当前阶段 `must_for_release = Y` 的 REQ 全部通过 ACPT
- 覆盖率检查通过
- migration 演练通过
- 审计与告警已启用
- 对应阶段的部署输入齐全

**S1 额外条件**：
- 4 条核心端到端链路可 UI 操作完成
- 多租户隔离验证通过
- 驾驶舱基础摘要可展示

**S2 额外条件**：
- S1 门禁全部维持
- 成交链路 E2E 可 UI 操作完成
- 高风险动作审批强制生效
- 审批/接管/回滚流程可走通
- 成交转化率可采集

**S3 额外条件**：
- S2 门禁全部维持
- 商业化闭环 E2E 可 UI 操作完成
- 欠费停服恢复链路可走通
- AI auto 模式可用且受控

**S4 额外条件**：
- S3 门禁全部维持
- 国际化全链路可走通
- 私有化部署+迁移+回滚可走通
- 六个结果指标全部可看板化

阶段门禁总表见第 14 节。
