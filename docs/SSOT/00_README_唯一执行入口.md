# MOY SSOT 唯一执行入口

## 1. 文档定位
`docs/SSOT/` 是 MOY 后续需求、设计、开发、测试、部署、迁移的唯一主文档体系。

自本次重构起：
- `docs/P0/` 不再是执行主干，仅保留为历史快照来源。
- `docs/FINAL/` 不再是执行主干，仅保留为终局边界历史快照。
- 所有新增或修订事实，必须回写到 `docs/SSOT/` 的事实归属文档，不得先改归档文档。

## 3. 单一事实源规则
### 3.1 事实只定义一次
- 需求事实只在 `03` 定义。
- 跟踪闭环只在 `04` 定义。
- 表结构、字段、索引、迁移顺序只在 `05` 定义。
- 状态与流转只在 `06` 定义。
- 权限、data scope、row scope、field scope、AI 边界只在 `07` 定义。
- API、Schema、错误码、WebSocket 事件只在 `08` 定义。
- 页面、按钮、字段、显隐、反馈、实时刷新点只在 `09` 定义。
- Agent、自动化、工具调用、审批、回滚、交接、集成执行只在 `10` 定义。
- 商业化规则、计费口径、套餐边界只在 `11` 定义；对象/状态/API/PAGE/TEST 只引用其他主文档 ID。
- TEST/ACPT/E2E、部署、回滚、租户初始化、升级迁移只在 `12` 定义。

### 3.2 阶段是字段，不是副本
所有主条目必须带：
- `introduced_in`
- `required_in`
- `release_scope`
- `status`

禁止再通过复制 `P0`、复制 `FINAL` 来表达阶段差异。

## 4. 阶段定义
| 阶段 | 定义 | 对应历史口径 |
| --- | --- | --- |
| S1 | 首期核心可交付 | 当前 P0 冻结实现级范围 |
| S2 | 首期完整版 / 增强版 | legacy P1 |
| S3 | 业务扩展版 | legacy P2 |

## 5. 阅读顺序
### 5.1 首次进入项目
1. 本文档 `00`
2. `01_产品范围与阶段地图.md`
3. `02_业务域与模块树.md`
4. `03_全量需求总表.md`
5. `04_需求追踪矩阵.md`
6. 再按实现主题进入 `05`~`12`

### 5.2 面向开发执行
1. 先在 `01` 确认目标阶段。
2. 在 `03` 过滤本次开发涉及的 REQ。
3. 在 `04` 拉齐闭环链路。
4. 根据实现主题先进入主文档 `05/06/07/08/09/10/11/12` 确认规则与索引。
5. 需要直接生成实现输入时，优先进入已进入“原子实现级”的子文档：
   - 数据与 migration：`05A`
   - API / Schema：`08A` + `08B`
   - 页面：`09A`
   - TEST / ACPT / 发布门禁：`12A`

### 5.3 面向代码生成工具
1. 阶段与范围：`01`
2. 模块与编码：`02`
3. 需求与闭环：`03` + `04`
4. 后端生成：`05` + `05A` + `06` + `07` + `08A` + `08B`
5. 前端生成：`07` + `08A` + `08B` + `09A`
6. AI/自动化/集成：`10`
7. 商业化：`11`
8. 测试、部署、迁移：`12` + `12A`

## 6. 裁决优先级
### 6.1 顶层规则
1. 本文档 `00` 负责解释 SSOT 使用规则与裁决机制。
2. 冲突优先由“事实归属文档”裁决，不按旧文档裁决。
3. `03` 与 `04` 是索引与闭环入口，不覆盖 `05`~`12` 的事实定义。
4. `13` 只记录历史映射与裁决结果，不反向生成事实。

### 6.2 主题裁决
| 冲突主题 | 胜出文档 |
| --- | --- |
| 阶段边界、范围上限 | `01` |
| 模块归属、模块编码、模块依赖 | `02` |
| 需求定义 | `03` |
| 需求闭环与 owner/status | `04` |
| 表结构与数据库迁移 | `05` |
| 状态机、合法/非法流转 | `06` |
| 权限、数据范围、AI 边界 | `07` |
| API、Schema、错误码、WebSocket | `08` |
| 页面与交互 | `09` |
| Agent、自动化、集成执行 | `10` |
| 商业规则与计费规则 | `11` |
| 测试、验收、部署、迁移 | `12` |
| 历史映射与旧口径解释 | `13` |

### 6.3 P0 / FINAL 冲突裁决
- 工程实现细节冲突：以 P0 口径为准，并在 SSOT 中固化。
- 终局范围上限冲突：以 FINAL 范围为准，并在 SSOT 中吸收。
- 编码体系冲突：以 P0 风格为准。
- FINAL 自相矛盾：不照抄，必须在 SSOT 中修正后再采用。

## 7. 统一编码规则
| 类型 | 规则 | 示例 |
| --- | --- | --- |
| 需求 | `REQ-{MOD}-{NNN}` | `REQ-CNV-003` |
| 页面 | `PAGE-{MOD}-{NNN}` | `PAGE-TK-002` |
| 接口 | `API-{MOD}-{NNN}` | `API-LM-005` |
| 数据表 | `TABLE-{MOD}-{NNN}` | `TABLE-SUB-001` |
| 权限 | `PERM-{MOD}-{ACTION}` | `PERM-CNV-TRANSFER` |
| 测试 | `TEST-{MOD}-{NNN}` | `TEST-BILL-002` |
| 验收 | `ACPT-{MOD}-{NNN}` | `ACPT-CT-003` |
| 状态机 | `SM-{entity_snake_case}` | `SM-ai_agent` |

统一模块码：
- 继承 P0：`AUTH ORG USR CM LM OM CNV TK TSK NTF CHN AI AUD SYS`
- 新增终局：`KB DASH AUTO QT CT ORD PLAN SUB BILL PAY INV CSM INT PLT I18N DEPLOY`

禁止继续使用下列旧口径作为主编码：
- `REQ-SM-*`
- `REQ-TM-*`
- `REQ-USER-*`
- `REQ-DB-*`
- `REQ-MA-*`
- `API-GET-*`
- `PAGE-001` 这类纯数字页号

## 8. 回写链路
统一回写顺序：
1. 定位事实归属文档。
2. 更新事实归属文档。
3. 更新受影响文档。
4. 更新 `04_需求追踪矩阵.md`。
5. 如涉及旧文档映射、裁决说明、别名迁移，再更新 `13_历史映射与归档说明.md`。

### 8.1 常见变更的回写路径
| 变更类型 | 回写链路 |
| --- | --- |
| 新增需求 | `03 -> 04 -> 05/06/07/08/09/12 -> 13(如有旧 ID 映射)` |
| 新增表/字段 | `05 -> 08(如接口变化) -> 09(如页面变化) -> 12 -> 04` |
| 新增状态流转 | `06 -> 08 -> 09 -> 12 -> 04` |
| 新增权限/数据范围 | `07 -> 08 -> 09 -> 12 -> 04` |
| 新增 API / Schema / WS | `08 -> 09 -> 12 -> 04` |
| 新增页面 | `09 -> 08 -> 07 -> 12 -> 04` |
| 新增 Agent/自动化/集成流程 | `10 -> 05/06/07/08/09/12 -> 04` |
| 商业化规则变更 | `11 -> 05/06/07/08/09/12 -> 04` |
| 部署/迁移变更 | `12 -> 05/08(如需) -> 13(如为迁移裁决)` |

## 9. 按阶段筛选开发范围
### 9.1 必须使用的过滤条件
- 开发范围：`introduced_in <= 当前阶段`
- 发布门禁：`required_in <= 当前阶段 AND must_for_release = Y`
- 实施看板：`release_scope` 包含当前阶段且 `status != validated`

### 9.2 推荐执行步骤
1. 在 `01` 确认当前开发阶段。
2. 在 `03` 过滤出目标阶段全部 REQ。
3. 在 `04` 拉出对应 TABLE / SM / PERM / API / PAGE / TEST / ACPT。
4. 分别进入 `05`~`12` 执行实现、联调、验收、迁移。
