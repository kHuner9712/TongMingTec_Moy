# MOY App S2 灰度前检查清单（12D）

## 1. 使用说明
- 本清单用于 S2 Release Candidate 灰度前最终检查，目标是把“人工复跑通过”升级为“可持续验证通过”。
- 执行原则：每一项都要有可复跑证据（命令输出、报告、截图或日志链接）。
- 结论口径：
  - 全部关键项通过：可进入灰度窗口。
  - 任一关键项失败：禁止灰度，先修复后复跑。

## 2. Gate 检查
| 检查项 | 命令 | 通过标准 | 证据 |
| --- | --- | --- | --- |
| S2 硬门禁总入口 | `cd frontend && npm run test:gate:s2` | 全部测试通过 | |
| S2 advanced acceptance（Chromium） | `cd frontend && npm run test:gate:s2:advanced` | 全部通过，无 skip 掩盖 | |
| 结果责任链专项 | `cd frontend && npm run test:gate:s2:responsibility` | 链路 A/B/C/D 全通过 | |

## 3. 数据库迁移与 Smoke
| 检查项 | 命令 | 通过标准 | 证据 |
| --- | --- | --- | --- |
| Migration smoke | `cd frontend && npm run test:smoke:migration` | 独立 smoke 库迁移成功，`migrations` 记录数 > 0 | |
| Test-support smoke | `cd frontend && npm run test:smoke:test-support` | 登录成功 + `ensure-secondary-tenant` 成功，且主/次租户不同 | |
| 夜间作业最近一次结果 | GitHub Actions `Nightly Smoke` | 最近一次 nightly 全绿 | |

## 4. 发布前配置核查
| 检查项 | 通过标准 | 证据 |
| --- | --- | --- |

| 回滚预案可执行 | 应用回滚步骤与 migration 回滚策略已确认并演练（至少文档级演练） | |

| 主链 | 抽查内容 | 通过标准 | 证据 |

## 7. 失败回退最小动作
1. 立即暂停灰度流量切入。
2. 回滚到上一个已验证版本（应用与配置同回滚单）。
3. 若涉及 schema 风险，执行已备案 migration 回滚策略。
4. 重新执行 `test:gate:s2` 与两项 smoke，恢复全绿后再申请灰度窗口。
