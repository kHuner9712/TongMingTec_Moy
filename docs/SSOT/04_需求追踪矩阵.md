# MOY 需求追踪矩阵

## 1. 文档定位
本文档是 MOY 唯一 RTM。所有 REQ 必须闭环映射到：

`REQ -> OBJ/TABLE -> SM -> PERM -> API -> PAGE -> TEST -> ACPT`

补充管理字段：
- `release_scope`
- `status`
- `owner`
- `implementation_stage`

## 2. RTM 规则
- 覆盖率要求：`REQ -> TABLE/API/PAGE/TEST/ACPT = 100%`。
- 对没有显式状态流转的需求，`SM` 可记为 `-`，但必须说明原因。
- 同一 REQ 可以映射多个 API、多个 PAGE、多个 TEST/ACPT。
- `owner` 取实现主责团队，不代表唯一协作方。

## 3. owner 约定
| owner | 含义 |
| --- | --- |
| platform-core | 平台基础、认证、组织、权限、系统配置 |
| crm | 客户、线索、商机主链 |
| service | 会话、工单、任务、通知、知识、看板 |
| ai | AI、Agent、审批、质检、回滚 |
| growth | 营销自动化 |
| revenue | 报价、合同、订单、订阅、账单、支付、发票、客户成功 |
| ecosystem | 渠道、多端、集成、开放平台 |
| infra | 国际化、私有化、迁移、运维 |

## 5. 覆盖率结论
| 维度 | 覆盖率 | 说明 |
| --- | --- | --- |
| REQ -> TABLE | 100% | 每条 REQ 均映射到至少一张主表或支撑表 |
| REQ -> API | 100% | 每条 REQ 均映射到实现接口 |
| REQ -> PAGE | 100% | 每条 REQ 均落到至少一个执行页面或控制台 |
| REQ -> TEST | 100% | 每条 REQ 均有测试 ID |
| REQ -> ACPT | 100% | 每条 REQ 均有验收 ID |

## 6. 使用规则
- 任何新增 REQ、TABLE、SM、PERM、API、PAGE、TEST、ACPT，都必须同步回写本矩阵。
- `status=validated` 的条目允许直接驱动实现；`status=specified/planned` 的条目必须先完成对应事实文档细化后再进入开发。
- `implementation_stage` 允许使用：
  - `Sx-implemented`
  - `Sx-implementation-ready`
  - `Sx-planned`
- 当 `05A/08A/08B/09A/12A` 已补齐实现级细节但代码尚未落地时，优先把条目标记为 `Sx-implementation-ready`，而不是继续停留在“规格未成形”的语义里。
