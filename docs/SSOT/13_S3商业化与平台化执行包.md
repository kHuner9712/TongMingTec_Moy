# MOY App S3 商业化与平台化执行包

## 1. 文档定位
- 文档目标：把 S3 从“模块名”收口为可执行工程路线，约束后续实现顺序、门禁与证据。
- 适用阶段：S2 RC（待灰度）之后，进入 S3 研发与发布门禁阶段。
- 约束：不新增模块码、不重命名既有模块；沿用 `PLAN/BILL/INV/INT/PLT` 与 `ORD/PAY/SUB` 扩展口径。
- 事实来源：`01/02/03/04/11/12C`；本文件是 S3 执行编排文档，不替代对象/API/状态机主文档。

## 2. S3 总目标
S3 的目标不是“把 S2 做大”，而是把 S2 成交后主链升级为“可商业化运营 + 可对外开放”的双闭环系统：

1. 商业化闭环：`订阅 -> 账单 -> 支付 -> 发票` 与 `欠费 -> 催缴 -> 停服 -> 恢复` 可持续运行。
2. 平台化闭环：`集成配置 -> 字段映射/事件订阅 -> Webhook/Open API 输出 -> 投递追踪` 可运营。
3. 与 S2 主链连续：不破坏 `OM->QT->CT->ORD->PAY->SUB->DLV->CSM`，S3 仅在该主链上加深商业能力与开放能力。

## 3. S3 双主轴（必须拆分执行）

### 3.1 商业化主轴
范围：`PLAN/BILL/INV + ORD/PAY/SUB 扩展`

- `PLAN`：套餐、增购、额度策略，提供“卖什么、怎么算、能用多少”的统一口径。
- `BILL`：账单生成、明细结算、欠费与催缴事件。
- `INV`：发票申请、开具、投递、下载。
- `ORD/PAY/SUB` 扩展：续费、对账、退款、停服与恢复、席位治理。

### 3.2 平台化主轴
范围：`INT/Webhook/Open API/PLT`

- `INT`：集成配置、集成流、字段映射、失败补偿。
- `Webhook`：事件订阅、签名校验、重试策略、投递追踪。
- `Open API`：API Key、开放接口、调用审计与限流口径。
- `PLT`：门户与多端工作台（首批聚焦运营控制台与客户自助入口，不先做完整多端矩阵）。

## 6. 主链对象关系（S3）

```text
S2 成交主链
OM -> QT -> CT -> ORD -> PAY -> SUB

S3 商业化扩展
PLAN(plan/addon/quota_policy)
  -> SUB(subscription/renewal)
  -> BILL(bill/bill_item)
  -> PAY(payment: bill/renewal scene)
  -> INV(invoice)

欠费治理链
BILL(overdue_event)
  -> NTF(催缴)
  -> SUB(suspension_record: suspend/resume)
  -> CSM(风险/续费动作)

平台化输出链
INT(integration/integration_flow/mapping)
  -> Webhook(webhook/webhook_delivery)
  -> Open API(api_client + audit)
  -> PLT(门户/多端消费)
```

## 7. S3 核心端到端链路（至少 4 条）

### 链路 A：订阅 -> 账单 -> 支付 -> 发票
1. 订阅在账期触发生成账单（含 bill_item）。
2. 支付成功回写账单状态。
3. 发票申请并开具，完成投递/可下载。
4. 门禁证据：`API-BILL-* + API-PAY-* + API-INV-*`、对应页面链路与 ACPT。

### 链路 B：欠费 -> 催缴 -> 停服 -> 恢复
1. 账单逾期写入 overdue 事件。
2. 自动催缴（通知 + 任务），超过阈值触发停服。
3. 欠费清零或人工审核后恢复订阅。
4. 门禁证据：逾期事件日志、停服审计日志、恢复事件日志。

### 链路 C：续费 -> 订单/账单 -> 支付 -> 生效
1. 发起续费（可由 CSM 或系统策略触发）。
2. 生成续费账单/支付场景。
3. 支付确认后续费生效，订阅账期与权益更新。
4. 门禁证据：续费状态机流转 + 订阅权益对比前后快照。

### 链路 D：集成配置 -> Webhook / API 输出
1. 配置集成凭证、事件订阅、字段映射。
2. 业务事件触发 Webhook 投递，失败重试可追踪。
3. 外部系统使用 Open API 拉取或推送业务对象。
4. 门禁证据：签名校验、重试日志、调用审计、隔离验证。

## 8. 与 S2 主链衔接点（不可破坏）

1. `ORD/PAY/SUB`：保留 S2 状态机与审批边界，仅新增 S3 场景（续费、对账、停服恢复）。
2. `DLV/CSM`：停服、恢复、续费结果要回写客户成功视图，避免“财务动作与经营动作脱节”。
3. `AUTO`：可消费 `bill.overdue`、`renewal.due` 等事件，但高风险动作继续走审批边界。
4. `DASH`：S3 新增商业指标必须标注口径（真实/代理/覆盖率）与数据质量说明。
5. `org_id`：所有新增对象、API、事件继续遵循多租户隔离，不开例外通道。

## 9. 批次拆分：最小可运行 vs 增强

### 9.1 第一批（最小可运行，先做）
- 商业化：`PLAN-001/002`、`BILL-001`、`INV-001`、`SUB-002`、`PAY-002（最小对账场景）`。
- 平台化：`INT-001`、`INT-003`、`INT-004（API Key + 基础开放接口）`。
- 要求：四条核心链路全部可跑通，且每条链路都有 API + 页面 + E2E 证据。

### 9.2 第二批（增强能力，后做）
- 商业化增强：`BILL-002/003` 深化、`INV-002`、复杂退款与冲正策略。
- 平台化增强：`INT-002` 复杂映射补偿、`PLT-001` 门户首版、`PLT-002/003/004` 多端能力。
- 要求：从“流程可跑”升级为“可运营、可观测、可治理”。

## 11. 建议先实现的第一批模块

1. `PLAN`（套餐/增购/额度）
2. `BILL`（账单生成 + 基础应收状态）
3. `SUB/PAY` 扩展（续费生效 + 对账最小链路）
4. `INV` 最小闭环（申请/开具）
5. `INT` 最小闭环（集成配置 + Webhook + Open API Key）

---
执行原则：先打通“能收费、能追款、能开票、能对外同步”的最小闭环，再补平台化深水区能力；不在 S3 初期并行铺开完整多端矩阵。
