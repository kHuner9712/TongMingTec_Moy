## 3. 统一冲突裁决记录
### 3.1 工程细节以 P0 为准
- 路径参数统一 `{id}`。
- `/` 仅重定向到 `/dashboard`。
- 商机结果只允许 `won/lost`。
- 会话权限必须拆分 `SEND/ACCEPT/TRANSFER/CLOSE`。
- 多租户隔离以 `org_id` 为核心。

### 3.2 终局范围以 FINAL 为准
- 知识库、看板、营销自动化、报价、合同、客户成功进入 S2。

- 国际化、私有化、字段权限、终局 Agent 规模化进入 S4。

### 3.3 编码统一
- 保留 P0 风格：`REQ/PAGE/API/TABLE/PERM/TEST/ACPT/SM`
- 淘汰 FINAL 风格：
  - `API-GET-CM-001`
  - `PAGE-001`
  - `REQ-SM-*`
  - `REQ-TM-*`
  - `REQ-USER-*`
  - `REQ-DB-*`
  - `REQ-MA-*`

- `payment` 与 `回款` 合并为单一 `payment` 对象。
- `invoice_request` 吸收进 `invoice`。
- `renewal_order` 吸收进 `renewal`。

- `FINAL` 中尚未成文或“待编写”的终局蓝图，不再以空白蓝图保留，统一落成 SSOT 主规格。

## S2 阶段重新定义裁决（2026-04-12）

### 裁决 1：S2 定义调整
- **旧口径**：S2 = 首期完整版 / 增强版，范围 = KB DASH AUTO QT CT CSM
- **新口径**：S2 = 成交与成交后衔接主干，范围 = KB DASH AUTO QT CT ORD PAY SUB CSM
- **裁决原因**：产品定位要求覆盖"获客→沟通→谈单→成交→售后→续费"全链路，S1 已完成前链路，S2 必须补齐成交与成交后衔接主干才能形成可运营闭环
- **影响文档**：01/02/03/04/05/05A/06/07/08/08A/08B/09/09A/10/11/12/12A

### 裁决 2：ORD/PAY/SUB 最小内核从 S3 提前到 S2
- **旧口径**：ORD/PAY/SUB 的 introduced_in/required_in = S3
- **新口径**：ORD/PAY/SUB 的最小内核 introduced_in/required_in = S2，完整商业化能力仍保留在 S3
- **裁决原因**：商机赢单→报价→合同→订单→付款→开通/订阅 是不可断裂的成交主链，如果订单/付款/订阅延迟到 S3，则 S2 的报价和合同无法形成交付闭环
- **S2 最小内核边界**：
  - ORD：订单创建（从合同/报价生成）、订单详情与明细、订单确认与激活、订单与付款的关联
  - PAY：付款记录创建（关联订单/合同）、付款状态跟踪、付款确认与订单激活的联动
  - SUB：从订单开通订阅、订阅基础状态管理、订阅与客户的关联
- **S3 保留能力**：
  - ORD：退款、完整履约
  - PAY：对账、退款、冲正
  - SUB：续费流程、停服与恢复、席位管理
