# MOY 品牌架构与多站点边界

## 1. 文档定位

本文档是 MOY 母品牌、产品矩阵、多站点域名、产品线归属的唯一事实源。

本文档负责裁决：

- MOY 母品牌是什么
- MOY App 是什么
- MOY GEO 是什么
- MOY API 是什么
- moy.com / app.moy.com / geo.moy.com / api.moy.com / docs.moy.com / status.moy.com 分别承担什么职责
- 当前 frontend/backend 属于哪条产品线
- 后续新增 GEO/API/官网能力时应该放在哪里

## 2. MOY 新总定位

MOY 是桐鸣科技旗下的企业 AI 增长与经营自动化平台。

MOY 不再等同于当前仓库里的 AI 客户经营系统。
当前仓库里的 frontend/backend 主体系统应命名为：

**MOY App - AI 原生客户经营系统**

## 3. 产品矩阵

### MOY Official

| 属性 | 内容 |
| --- | --- |
| 域名 | moy.com |
| 定位 | 品牌官网、产品矩阵入口、商业转化入口 |
| 主要用户 | 潜在客户、合作伙伴、投资/合作对象 |
| 当前工程归属 | sites/official |

### MOY App

| 属性 | 内容 |
| --- | --- |
| 域名 | app.moy.com |
| 定位 | AI 原生客户经营系统 |
| 主要用户 | 销售团队、客服团队、交付团队、客户成功团队 |
| 当前工程归属 | frontend / backend 主体系统 |
| 功能边界 | 客户、线索、商机、会话、工单、报价、合同、订单、支付、订阅、交付、客户成功、AI Agent、审批、接管、回滚、自动化、经营驾驶舱 |

> 注意：MOY App 是 MOY 母品牌下的一条产品线，不再等同于全部 MOY。

### MOY GEO

| 属性 | 内容 |
| --- | --- |
| 域名 | geo.moy.com |
| 定位 | AI 搜索增长与品牌可见度服务 |
| 主要用户 | 需要 AI 搜索曝光、品牌推荐、内容资产建设的企业 |
| 当前工程归属 | sites/geo + docs/GEO |
| 第一阶段形态 | 营销页 + 诊断表单 + 服务化交付 |

> 注意：不直接塞入 MOY App 主菜单。

### MOY API

| 属性 | 内容 |
| --- | --- |
| 域名 | api.moy.com |
| 定位 | 多模型 API 网关 / API 中转站 / 开发者平台 |
| 主要用户 | 开发者、AI 应用团队、企业技术团队 |
| 当前工程归属 | sites/api + docs/API_HUB |
| 第一阶段形态 | 营销页 + 开发者入口 + API Key + 用量统计 + 模型列表 + 调用日志 |

> 注意：不要包装成低价 API 倒卖站。

## 4. 域名职责

| 域名 | 职责 |
| --- | --- |
| moy.com | 品牌官网 / 产品矩阵入口 |
| app.moy.com | MOY App 主系统 |
| geo.moy.com | MOY GEO 服务站 |
| api.moy.com | MOY API 平台 |
| docs.moy.com | 文档中心（后期） |
| status.moy.com | 服务状态页（后期） |

禁止使用：**moy.moy.com**

## 5. 共享底座

未来三条产品线逐步共享：

- 统一账号
- 统一组织/租户
- 统一权限
- 统一订阅/计费
- 统一 AI Gateway
- 统一审计
- 统一日志
- 统一通知
- 统一工单/支持

但第一阶段不强行一次性统一。

## 6. 独立边界

必须明确：

- MOY App 是业务经营系统
- MOY GEO 是增长服务
- MOY API 是模型调用基础设施
- 三者前台入口独立，底层能力逐步共用
- 不允许把 GEO 和 API 简单作为当前 App 的普通菜单功能处理

## 8. 第一阶段 MVP

第一阶段只做：

- moy.com 品牌官网骨架
- geo.moy.com GEO 服务站骨架
- api.moy.com API 平台站骨架
- 当前系统命名改为 MOY App
- GEO 服务化交付文档
- API MVP 技术文档
- 暂不做完整统一账号
- 暂不做完整 billing
- 暂不做完整 SaaS 化 GEO
- 暂不做复杂 monorepo 大迁移

## 9. 不做事项

明确不做：

- 不推倒重写当前 MOY App
- 不把 moy.com 直接作为登录系统
- 不使用 moy.moy.com
- 不把 API 中转包装成低价倒卖 API
- 不做虚假 GEO、垃圾内容批量发布、伪造媒体报道、伪造客户案例
- 不在第一阶段做复杂 monorepo 大迁移
- 不把 GEO/API 默认写入 MOY App 主菜单

## 11. 裁决规则

- 品牌与多站点边界冲突时，以本文档为准
- MOY App 内部业务模块仍以 01~13 文档为准
- 新增 GEO/API 相关需求时，必须先判断是否属于独立产品线，不得默认写入 MOY App 业务模块
- 涉及官网、GEO、API、产品矩阵、域名、命名体系的变更，必须先更新本文档
