# MOY 品牌架构与多站点边界

## 1. 文档定位

本文档是 MOY 母品牌架构、多产品线边界与多站点域名的唯一事实源。

本文档负责定义：

- MOY 母品牌的定位与边界
- 四条产品线（MOY App / MOY GEO / MOY API / MOY Official）的职责与形态
- 六个子域名（`moy.com` / `app.moy.com` / `geo.moy.com` / `api.moy.com` / `docs.moy.com` / `status.moy.com`）的职责边界
- 当前仓库中已有系统的归属
- 跨产品线共享底座与独立边界的划分
- 第一阶段 MVP 范围与明确不做事项
- 后续新增产品线的工程边界与准入规则
- 品牌与多站点边界冲突时的裁决规则

本文档 `introduced_in=S2`，`required_in=S2`，`status=implemented`。与 `00_README_唯一执行入口.md` 的目录总览保持同步。

## 2. MOY 新总定位

### 2.1 母品牌定义

**MOY（墨言）是桐鸣科技旗下的企业 AI 增长与经营自动化平台。**

MOY 不再等同于单一的 AI 客户经营系统，而是母品牌。当前仓库里的主体业务系统应命名为 **MOY App**。

### 2.2 品牌核心主张

| 维度 | 定义 |
| --- | --- |
| 所属主体 | 桐鸣科技 |
| 品牌定位 | 企业 AI 增长与经营自动化平台 |
| 产品形态 | 多产品线母品牌，每条产品线有独立域名、独立入口、独立交付节奏 |
| 服务对象 | B2B 企业客户 |
| 品牌调性 | 结果交付型、AI 原生、工程技术驱动 |
| 价值主张 | 让企业用 AI 实现可度量、可追责、可复利的增长与经营自动化 |

### 2.3 命名规范

| 场景 | 正确用法 | 禁止用法 |
| --- | --- | --- |
| 指代整个品牌 | MOY / MOY 母品牌 | — |
| 指代客户经营系统 | MOY App | 继续单独叫"MOY"（模糊指代） |
| 指代搜索增长服务 | MOY GEO | "MOY 的 GEO 功能"（它是独立产品线，不是功能） |
| 指代 API 网关 | MOY API | "MOY 的 API 模块"（同上） |
| 指代品牌官网 | MOY Official / moy.com | — |

### 3.1 MOY App 详述

MOY App 是 MOY 旗下最核心的产品线，承载完整的客户经营业务闭环。

| 维度 | 定义 |
| --- | --- |
| 一句话定位 | AI 运行时驱动的结果交付型客户经营系统 |
| 核心主链 | 获客 → 沟通 → 商机 → 报价 → 合同 → 订单 → 付款 → 订阅 → 交付 → 验收 → 客户成功 |
| AI 治理 | Agent 运行时 + 审批中心 + 接管中心 + 回滚中心 |
| 阶段状态 | S1 基本完成，S2 成交后衔接主干已实现、待通过阶段门禁 |
| 技术栈 | React 18 + NestJS 10 + PostgreSQL + TypeORM |
| 本地端口 | 前端 `:5173`，后端 `:3001` |

MOY App 的内部业务模块、状态机、API、页面、测试仍以 01~13 号文档为准，不受本文档对品牌架构的重新定义影响。

| 维度 | 定义 |
| --- | --- |

| 覆盖平台 | ChatGPT、豆包、Kimi、通义千问、Perplexity、Google AI Overviews、Bing Copilot 等 |
| 第一阶段交付方式 | 服务化交付为主，不做复杂 SaaS 产品形态 |
| 第一阶段核心 | 品牌可见度诊断报告 + 基础优化建议 + 轻量客户看板 |
| 与 MOY App 关系 | 前台入口独立，菜单不互嵌；长期可通过客户 360 视图消费 GEO 数据 |

### 3.3 MOY API 详述

MOY API 是独立产品线，不是 MOY App 的一个技术模块。

| 维度 | 定义 |

## 5. 共享底座

三条产品线长期共享以下底层能力。当前阶段各自独立运行，共享底座在 S4 阶段统一建设。

| 共享底座 | 说明 | 建设阶段 |
| --- | --- | --- |
| 统一账号 | 一个 MOY 账号通行所有产品线（SSO / OAuth） | S4 |
| 统一组织/租户 | `org_id` 作为跨产品线的客户身份标识 | S4 |
| 统一权限 | RBAC + data scope 跨产品线一致 | S4 |
| 统一订阅/计费 | 一个订阅覆盖多条产品线的用量与账单 | S4 |
| 统一 AI Gateway | MOY API 作为所有产品线的模型调用基础设施 | S4 |
| 统一审计 | 跨产品线的关键操作审计日志汇聚 | S4 |
| 统一日志 | 跨产品线的应用日志采集与检索 | S4 |
| 统一通知 | 跨产品线的站内信、邮件、短信通知中心 | S4 |
| 统一工单/支持 | 客户跨产品线问题时，统一工单入口与流转 | S4 |

共享底座的建设原则：

1. 不在 MVP 阶段提前建设。共享底座的建设成本远高于短期收益，必须等各产品线各自验证 PMF 后再统一。
2. 共享底座只做能力收敛，不做业务耦合。三条产品线的业务数据、业务逻辑、业务状态机各自独立。
3. 共享底座通过 API / SDK / SSO 协议集成，不做共享数据库、不做共享 ORM、不做跨库事务。

## 6. 独立边界

三条产品线的前台入口、业务逻辑、数据模型、交付节奏各自独立。

| 维度 | MOY App | MOY GEO | MOY API |
| --- | --- | --- | --- |
| 本质 | 业务经营系统 | 增长服务 | 模型调用基础设施 |
| 前台入口 | `app.moy.com` | `geo.moy.com` | `api.moy.com` |
| 核心用户 | 业务团队 | 市场/品牌团队 | 开发团队 |
| 数据模型 | 客户/线索/商机/合同/订单等 | 品牌/平台/诊断/报告等 | API Key/模型/用量/日志等 |
| 交付节奏 | 按 S1~S4 阶段推进 | 服务化 → 产品化 | MVP → 平台化 |
| 计费模型 | 按席位 + 订阅周期 | 按诊断次数 / 项目制 | 按 token 用量 |

工程纪律：

1. MOY App 是业务经营系统，MOY GEO 是增长服务，MOY API 是模型调用基础设施 — 三者前台入口独立，底层能力逐步共用。
2. **不允许把 GEO 和 API 简单作为当前 App 的普通菜单功能处理。** 它们各自有独立的产品定位、用户群体和商业模型。
3. 跨产品线的能力调用只能通过 API / SDK / SSO 协议，不允许跨产品线直接读写数据库。
4. 新增产品线必须先通过本文档的准入判断（见第 11 节裁决规则），不得在未定版前直接开工。

| `docs/_archive/` | 历史归档 | P0/FINAL 历史快照 |

### 7.2 新增目录的工程约束

后续新增 `official-site/`、`geo-site/`、`api-site/` 等正式站点目录时：

1. 不应破坏现有 `frontend/` `backend/` 结构。
2. 不应要求 MOY App 的 `App.tsx` 或 `app.module.ts` 引入 GEO/API 的路由或模块。
3. 不应在 MOY App 的构建配置中加入 GEO/API 的依赖。
4. 各自独立 `package.json`、独立 `node_modules`、独立构建产物。

## 9. 不做事项

以下事项在**全阶段明确不做**，属于品牌与产品的硬边界：

| 不做事项 | 原因 |
| --- | --- |
| 推倒重写当前 MOY App | 现有系统已投入大量工程资源，功能可用、架构合理，品牌重定位不需要代码重写 |
| 把 `moy.com` 直接作为登录系统 | `moy.com` 是品牌官网，登录入口在各子站 |
| 使用 `moy.moy.com` 等变体域名 | 域名体系已定版，不再引入变体 |
| 把 API 中转包装成低价倒卖 API | MOY API 的定位是价值型网关（统一接入、管理、治理），不是低价转售 |
| 不做虚假 GEO | 禁止虚假品牌曝光、垃圾内容批量发布、伪造媒体引用、伪造客户案例 |
| 不在第一阶段做复杂 monorepo 大迁移 | 三条产品线各自独立构建，暂不做 turborepo/nx/lerna 改造 |
| 不在 MOY App 业务菜单中嵌入 GEO/API | 不符合"独立站点、独立入口"原则 |

### 10.3 技术决策（已裁决）

| 决策项 | 裁决 | 理由 |

| MOY GEO 技术栈 | React + Vite + TypeScript（同 MOY App 前端栈） | 复用团队技能，降低学习成本；MVP 不引入新技术栈 |
| MOY API 后端技术栈 | Express + TypeScript（轻量 Node.js） | MVP 不需要 NestJS 复杂度；Express 满足网关 + 管理需求 |
| 仓库策略 | 当前仓库新增子目录（`geo/` `api-gateway/` `landing/`） | 三产品线同属 MOY 品牌，统一仓库便于代码索引；暂不拆分独立仓库 |
| 构建策略 | 各自独立 `package.json` + 独立构建 | 不做 monorepo 统一构建；各站点独立 dev/build/deploy |

## 11. 裁决规则

### 11.1 品牌与多站点边界冲突

发生以下冲突时，**以本文档（14）为准**：

- 品牌命名、产品线归属、域名职责的冲突
- "某个能力应该放在哪个产品线"的归属争议
- "是否应该新增一条产品线"的准入判断

### 11.2 MOY App 内部业务模块

MOY App 内部业务模块的冲突裁决仍以 01~13 号文档为准，包括：

- 模块归属与模块编码 → `02`
- 状态机与合法流转 → `06`
- 权限与数据范围 → `07`
- API 与 Schema → `08`
- 页面与交互 → `09`
- Agent 与自动化 → `10`
- 商业规则 → `11`
- 测试与部署 → `12`

### 11.3 新增 GEO/API 相关需求

提出新增 GEO 或 API 相关需求时，必须先通过以下判断：

1. 该需求是否属于独立产品线（GEO 或 API）？ → 如果是，不得写入 MOY App 业务模块（01~12 号文档）。
2. 该需求是否跨产品线（如"GEO 数据在 App 中展示"）？ → 如果是，必须先在本文档中登记跨产品线集成方案，再进入对应产品线的需求文档。
3. 该需求是否本质上是 MOY App 的能力增强？ → 如果是，按 MOY App 正常需求流程走（03 → 04 → 05~12）。

**禁止行为**：未经上述判断，直接将 GEO/API 需求默认写入 MOY App 的业务模块或前端路由。

### 11.4 与 00 号文档的关系

- `00` 负责解释 SSOT 体系的使用规则与阅读顺序。
- `14` 负责定义母品牌架构与多站点边界。
- 两者不存在上下位冲突：00 是"怎么用 SSOT"，14 是"MOY 品牌长什么样"。
- 涉及品牌级冲突时，14 的事实优先于其他文档中未经更新的旧表述。

## 12. 工程纪律

1. 任何新增产品线不得绕开本文档的准入判断直接侵入 MOY App 的代码、菜单或数据库。
2. 任何"多个产品线能不能合并/统一入口"的提议，必须先对照本文档第 5 节（共享底座）和第 6 节（独立边界）进行评判。
3. 跨产品线的能力建设必须先落设计文档，再开始编码。
4. 本文档的修订必须同步更新 `00_README_唯一执行入口.md` 的目录总览。

## 13. 简短 FAQ

**Q：MOY App 和之前的 MOY 是一回事吗？**

A：是的。本文档生效前，所有文档中的"MOY"指的就是现在的 MOY App。这次不是改产品，是把"MOY"升级为母品牌，让未来 GEO 和 API 有清晰的归属。

**Q：我写 MOY App 代码的时候需要改什么？**

A：业务代码不需要改。元数据层面（package.json description、页面标题、Swagger 标题）已改为 MOY App。核心业务逻辑、模块码、API 路由、数据库 schema 不受任何影响。

**Q：GEO 和 API 的代码在哪儿？**

A：`geo/` 和 `api-gateway/` 目录下已有 MVP 骨架。它们是独立项目，不会混入 MOY App 的 `frontend/` `backend/`。

**Q：GEO 和 API 什么时候能用？**

A：按第 10 节路线图分阶段推进。GEO 和 API 的节奏不阻塞 MOY App 的 S2 门禁 → S3 推进。
