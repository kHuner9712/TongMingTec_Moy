# MOY 结果责任链 E2E 验证设计与最小实现（12B）

## 1. 文档定位
本文档用于证明 MOY 不是模块堆砌，而是可验证的“结果责任链”系统。

- 对齐对象：SSOT、后端模块装配、前端入口、E2E 骨架
- 本轮目标：提供可运行最小验证，不做第二套测试世界
- 对应实现：`frontend/e2e/result-responsibility-chain.spec.ts`

## 2. 本轮覆盖链路

### 链路 A
线索 -> 转客户/商机 -> 商机赢单 -> 创建报价 -> 审批 -> 创建合同 -> 签署

### 链路 B
合同 -> 订单 -> 付款 -> 订单激活 -> 订阅开通

### 链路 C
订阅/交付 -> 纳入客户成功 -> 健康度评估 -> 续费提醒

### 链路 D
会话 -> AI 建议 -> 人工确认 -> 发送 -> 转工单 -> 工单关闭

## 3. 依赖矩阵（API / 状态机 / 权限）

### 3.1 链路 A
- 模块：`LM -> OM -> QT -> CT`
- 关键 API：
  - `POST /leads`
  - `POST /leads/:id/assign`
  - `POST /leads/:id/follow-ups`
  - `POST /leads/:id/convert`
  - `POST /opportunities/:id/stage`
  - `POST /opportunities/:id/result`
  - `POST /quotes`
  - `POST /quotes/:id/submit-approval`
  - `POST /quotes/:id/approve`
  - `POST /contracts`
  - `POST /contracts/:id/submit-approval`
  - `POST /contracts/:id/approve`
  - `POST /contracts/:id/sign`
- 状态机：
  - `SM-lead`: `new -> assigned -> following -> converted`
  - `SM-opportunity`: `discovery -> qualification -> proposal -> negotiation`，结果 `won/lost`
  - `SM-quote`: `draft -> pending_approval -> approved`
  - `SM-contract`: `draft -> pending_approval -> approved -> signing`
- 权限：
  - `PERM-LM-*`, `PERM-OM-*`, `PERM-QT-*`, `PERM-CT-*`

### 3.2 链路 B
- 模块：`CT -> ORD -> PAY -> SUB`（DealChain 事件编排）
- 关键 API：
  - `POST /orders/from-contract`
  - `POST /orders/:id/confirm`
  - `POST /art/approvals/:id/approve`
  - `POST /payments`
  - `POST /payments/:id/process`
  - `POST /payments/:id/succeed`
  - `POST /subscriptions`（由 DealChain 事件触发创建）
- 状态机：
  - `SM-order`: `draft -> pending_approval -> confirmed -> active`
  - `SM-payment`: `pending -> processing -> pending_approval -> succeeded`
  - `SM-subscription`: `trial -> active`
- 权限：
  - `PERM-ORD-MANAGE`, `PERM-ORD-ACTIVATE`, `PERM-PAY-*`, `PERM-SUB-MANAGE`, `PERM-AI-APPROVE`

### 3.3 链路 C
- 模块：`SUB -> DLV -> CSM`（`DlvEventHandler` + `CsmEventHandler`）
- 关键 API：
  - `GET /deliveries/by-order/:orderId`
  - `POST /deliveries/:id/risks`
  - `POST /deliveries/:id/outcomes`
  - `POST /deliveries/:id/status`
  - `POST /deliveries/:id/acceptances`
  - `GET /csm/health/:customerId`
  - `GET /csm/plans`
  - `POST /contracts/expire-check`
- 状态机：
  - `SM-delivery`: `draft -> active -> ready_for_acceptance -> accepted -> closed`
  - `SM-success-plan`: `draft -> active`（及后续流转）
- 权限：
  - `PERM-DLV-*`, `PERM-CSM-*`, `PERM-CT-MANAGE`

### 3.4 链路 D
- 模块：`CNV -> AI -> TK`
- 关键 API：
  - `GET /conversations`
  - `POST /ai/smart-reply`
  - `GET /ai/tasks/:id`
  - `POST /conversations/:id/messages`
  - `POST /conversations/:id/tickets`
  - `POST /tickets/:id/assign`
  - `POST /tickets/:id/start`
  - `POST /tickets/:id/resolve`
  - `POST /tickets/:id/close`
- 状态机：
  - `SM-conversation`: `queued -> active -> closed`
  - `SM-ticket`: `pending -> assigned -> processing -> resolved -> closed`
- 权限：
  - `PERM-CNV-*`, `PERM-AI-EXECUTE`, `PERM-TK-*`

## 4. 测试数据准备方案（不引入第二套世界）

### 4.1 通用原则
- 使用真实业务 API 造数（线索、商机、报价、合同、订单、付款、订阅、交付、工单）
- 使用系统既有审批中心完成关键审批动作
- 使用唯一前缀（`runTag`）避免污染与冲突

### 4.2 各链路最小准备
- 链路 A：从 `lead` 开始构造，按状态机推进到 `contract.signing`
- 链路 B：复用链路 A 的合同，走订单/付款审批闭环
- 链路 C：复用链路 B 的订单/订阅，验证交付与 CSM 联动
- 链路 D：优先复用现有会话数据；若无会话，显式标注卡点

## 5. 关键断言点

### 5.1 链路 A 断言
- 线索状态按 `new -> assigned -> following -> converted` 流转
- 转化输出 `customerId` 与 `opportunityId`
- 商机结果为 `won`
- 报价审批后状态进入 `approved/accepted`
- 合同审批并签署后状态为 `signing/active`

### 5.2 链路 B 断言
- 订单确认必须先进入 `pending_approval`
- 审批通过后订单为 `confirmed`
- 付款成功需经过审批，最终 `succeeded`
- 付款成功后订单自动 `active`
- 订阅被自动开通并关联 `orderId`

### 5.3 链路 C 断言
- 订单激活后存在交付单（`by-order` 可查询）
- 交付状态可推进到 `ready_for_acceptance -> accepted -> closed`
- 交付验收后可触发 CSM 健康度评估数据
- 成功计划可按客户查询
- 合同到期检查 API 可触发（续费提醒链路占位）

### 5.4 链路 D 断言
- 可提交 AI 建议任务并查询任务状态
- 人工确认后可发送消息
- 可由会话转工单
- 工单最终可关闭
- 若无会话可供测试，必须显式输出“缺会话自举 API”卡点

## 7. 本轮真实结论分级（代码事实）

### 7.1 已打通（后端主链 + E2E 骨架）
- 链路 A
- 链路 B
- 链路 C

### 7.2 条件打通（受测试数据可得性影响）
- 链路 D
  - 在存在可用会话数据时可跑通到工单关闭
  - 无会话时显式标注阻塞，不做静默跳过

### 7.3 当前明确卡点
- `缺数据对象`：无可用会话时无法自举链路 D
- `缺 API`：缺少标准化“创建/注入入站会话”测试接口（目前只能依赖现有会话）
- `外部依赖不确定`：AI 任务可能受运行时环境影响（失败时改为人工兜底消息并记录卡点）

## 8. 后续缺口清单
- 增加受控测试入口：入站会话创建 API（仅测试/受权限控制）
- 为链路 D 增加稳定种子数据机制，避免环境依赖
- 为“续费提醒”补充可断言对象（当前通过合同到期检查触发作为占位）
- 在发布门禁中加入“结果责任链 E2E”独立通过项
