## 一、文档目标

本文档作为 MOY 终局版未决事项的**权威清零基线**，用于：

1. **汇总所有未决事项**：整合 P0 文档体系中所有"待决策"、"待验证"、"待确认"项
2. **定义清零规则**：为每项未决事项提供默认规则、可配置参数、推荐取值范围
3. **明确决策依据**：记录每项决策的设计理由和后续验证方法
4. **标记清零状态**：跟踪所有未决事项直至清零
5. **消除开发阻塞**：确保开发团队无需等待决策即可按默认规则推进

## 三、术语定义

### 3.1 未决事项类型

| 类型       | 定义                                       | 示例                                       |
| ---------- | ------------------------------------------ | ------------------------------------------ |
| 待决策     | 需产品/业务方确认的产品功能决策            | 是否支持客户合并功能                       |
| 待验证     | 需通过访谈/测试验证的业务假设              | 销售人员每日处理线索数量                   |
| 待确认     | 需外部方（法务/第三方）确认的事项          | 数据存储位置合规要求                       |
| 待预研     | 需技术预研验证的技术方案                   | WebSocket并发性能测试                      |

### 3.2 清零状态

| 状态       | 定义                                       |
| ---------- | ------------------------------------------ |
| ✅ 已清零  | 已有明确决策，可按规则执行                 |
| 🔄 清零中  | 已有默认规则，待后续验证确认               |
| ⏳ 待清零  | 尚未确定默认规则，需优先处理               |

### 3.3 决策优先级

| 优先级     | 定义                                       | 处理时限     |
| ---------- | ------------------------------------------ | ------------ |
| P0-阻塞    | 阻塞首期开发，必须立即决策                 | 开发启动前   |
| P1-重要    | 影响核心功能，需在相关模块开发前决策       | 模块开发前   |
| P2-一般    | 影响用户体验或扩展功能，可后续迭代调整     | 上线前       |
| P3-低优    | 影响较小，可长期观察后决策                 | 持续迭代     |
