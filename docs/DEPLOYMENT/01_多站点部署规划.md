# MOY 多站点部署规划

## 1. 文档用途

本文档为 MOY 产品矩阵的全部站点制定部署方案。提供两套选型方案（平台托管 vs VPS 自建），供技术负责人根据当前资源和长期规划做决策。

**注意：本文档仅作规划，暂不执行部署。**

适用对象：DevOps、技术负责人、后端架构。

## 2. 站点总览

| 域名 | 用途 | 工程目录 | 类型 | 端口(本地) |
| --- | --- | --- | --- | --- |
| `moy.com` | 品牌官网 | `sites/official` | 纯静态 | 5175 |
| `geo.moy.com` | GEO 服务站 | `sites/geo` | 纯静态 | 5176 |
| `api.moy.com` | API 开发者平台 | `sites/api` | 纯静态 | 5177 |
| `app.moy.com` | MOY App 前端 | `frontend` | 纯静态 | 5173 |
| `api.app.moy.com` | MOY App 后端 API | `backend` | NestJS 服务 | 3001 |
| `docs.moy.com` | 文档中心 | — | 后期 | — |
| `status.moy.com` | 服务状态页 | — | 后期 | — |

> MVP 阶段仅前三项（brand/geo/api）+ 后端是硬需求；app.moy.com 优先级次于前三项。

## 3. 方案 A：Vercel / Netlify 托管（快速上线）

### 3.1 适用场景

- 只有纯静态站点，没有运维人力
- 需要快速上线、自动构建、免费 SSL
- 对自定义域名和 HTTPS 有需求但不想自己管理
- 后端可以单独托管到 Railway / Render

### 3.2 站点配置

| 站点 | 平台 | 仓库根目录 | Build Command | Output Directory | 框架预设 |
| --- | --- | --- | --- | --- | --- |
| `moy.com` | Vercel | `sites/official` | `npm run build` | `dist` | Vite |
| `geo.moy.com` | Vercel | `sites/geo` | `npm run build` | `dist` | Vite |
| `api.moy.com` | Vercel | `sites/api` | `npm run build` | `dist` | Vite |
| `app.moy.com` | Vercel | `frontend` | `npm run build` | `dist` | Vite |

每个站点在 Vercel 中是一个独立的 Project，设置独立的：

- **Root Directory**（对应上表）
- **Framework Preset**：Vite
- **Install Command**：`npm install`

### 3.3 环境变量

每个站点的 Vercel Project 需要设置环境变量：

| 站点 | 变量名 | 说明 | 示例 |
| --- | --- | --- | --- |
| `geo.moy.com` | `VITE_GEO_LEAD_ENDPOINT` | 线索表单后端 | `https://api.app.moy.com/api/geo/leads` |
| `app.moy.com` | `VITE_API_BASE_URL` | 后端 API 地址 | `https://api.app.moy.com/api/v1` |

### 3.4 域名绑定

在 Vercel Project → Settings → Domains 中绑定：

```
moy.com  → sites/official Project
www.moy.com → redirect to moy.com
geo.moy.com → sites/geo Project
api.moy.com → sites/api Project
app.moy.com → frontend Project
```

Vercel 自动提供 SSL 证书（Let's Encrypt）。

### 3.5 后端部署

`backend`（NestJS）不能部署到 Vercel（不是纯静态）。推荐：

| 平台 | 说明 | 价格 |
| --- | --- | --- |
| [Railway](https://railway.app) | 一键部署 Node.js，内建 PostgreSQL | $5/月起 |
| [Render](https://render.com) | Web Service，支持 Docker | $0 免费层 |
| [Fly.io](https://fly.io) | 全球边缘部署 | $0 免费层 |
| 阿里云 ECS / 腾讯云 CVM | 国内部署，低延迟 | 按量付费 |

环境变量（Railway / Render 面板配置）：

```bash
NODE_ENV=production
PORT=3001
DB_HOST=xxx
DB_PORT=5432
DB_NAME=moy_app
DB_USERNAME=moy
DB_PASSWORD=xxx
JWT_SECRET=xxx
```

### 3.6 CORS 配置

后端 `backend/src/main.ts` 或 NestJS CorsOptions 需配置：

```typescript
app.enableCors({
  origin: [
    "https://moy.com",
    "https://www.moy.com",
    "https://app.moy.com",
    "https://geo.moy.com",
    "https://api.moy.com",
  ],
  methods: ["GET", "POST", "PUT", "PATCH", "DELETE"],
  credentials: true,
});
```

**注意**：Vite dev server（localhost:5173）仅在开发时加入 origin。生产环境不含 `localhost`。

### 3.7 方案 A 优缺点

| 优点 | 缺点 |
| --- | --- |
| 零运维，自动 SSL，自动 CDN | 站点分散在多个 Vercel Project，不统一管理 |
| 免费额度足够 MVP 阶段使用 | 后端需额外找平台 |
| 部署时间 < 30 分钟 | 国内访问可能慢（Vercel 服务器在海外） |
| 自动 git push 触发部署 | 自定义 Nginx 规则不支持 |

## 4. 方案 B：VPS + Nginx 多站点（统一运维）

### 4.1 适用场景

- 有运维人力，需要精细控制 Nginx 路由
- 希望所有站点在一台服务器上管理
- 需要自定义日志、防火墙、SSL 策略
- 国内部署，低延迟

### 4.2 服务器建议

| 规格 | 推荐 |
| --- | --- |
| CPU | 2 核+ |
| 内存 | 4 GB+ |
| 系统 | Ubuntu 22.04 LTS |
| 用途 | 静态文件 + Node.js 进程 + Nginx 反向代理 |

### 4.3 目录规划

```
/var/www/moy/
├── official/          # sites/official → dist/
├── geo/               # sites/geo → dist/
├── api/               # sites/api → dist/
├── app/               # frontend → dist/
└── backend/           # backend 源码 + node_modules
```

构建后将各站点的 `dist/` 目录内容复制到对应路径。

### 4.4 Nginx 配置示例

```nginx
# /etc/nginx/sites-available/moy

# ---------- moy.com + www redirect ----------
server {
    listen 80;
    server_name moy.com www.moy.com;

    root /var/www/moy/official;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    # 静态资源缓存
    location /assets/ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }
}

# ---------- geo.moy.com ----------
server {
    listen 80;
    server_name geo.moy.com;

    root /var/www/moy/geo;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /assets/ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }
}

# ---------- api.moy.com ----------
server {
    listen 80;
    server_name api.moy.com;

    root /var/www/moy/api;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }
}

# ---------- app.moy.com ----------
server {
    listen 80;
    server_name app.moy.com;

    root /var/www/moy/app;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }
}

# ---------- api.app.moy.com → NestJS backend ----------
server {
    listen 80;
    server_name api.app.moy.com;

    location / {
        proxy_pass http://127.0.0.1:3001;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;

        # 超时设置（AI 接口可能较慢）
        proxy_read_timeout 120s;
        proxy_send_timeout 120s;
    }
}
```

### 4.5 HTTPS（Let's Encrypt）

```bash
# 安装 certbot
sudo apt install certbot python3-certbot-nginx

# 为所有域名申请证书
sudo certbot --nginx -d moy.com -d www.moy.com
sudo certbot --nginx -d geo.moy.com
sudo certbot --nginx -d api.moy.com
sudo certbot --nginx -d app.moy.com
sudo certbot --nginx -d api.app.moy.com

# 自动续期
sudo certbot renew --dry-run
```

### 4.6 后端进程管理（PM2）

```bash
npm install -g pm2

cd /var/www/moy/backend
pm2 start dist/main.js --name moy-backend --env production
pm2 save
pm2 startup  # 开机自启
```

### 4.7 方案 B 优缺点

| 优点 | 缺点 |
| --- | --- |
| 所有站点统一管理，Nginx 精细路由 | 需要运维能力（服务器管理、Nginx、SSL） |
| 国内部署低延迟 | 需要付费服务器（VPS ¥50-200/月） |
| 后端和静态文件在同一台机器，通信零延迟 | 服务器故障影响全部站点 |
| 可自定义日志、防火墙、监控 | 无自动扩容 |

## 5. 域名 DNS 配置

### 5.1 DNS 记录

| 记录类型 | 名称 | 值 | 说明 |
| --- | --- | --- | --- |
| A | `@` (moy.com) | VPS IP / Vercel | 根域名 |
| CNAME | `www` | `moy.com` | www 重定向 |
| CNAME | `geo` | VPS IP / `cname.vercel-dns.com` | GEO 子域名 |
| CNAME | `api` | VPS IP / `cname.vercel-dns.com` | API 子域名 |
| CNAME | `app` | VPS IP / `cname.vercel-dns.com` | App 子域名 |
| A | `api.app` | VPS IP（方案 B） | 后端 API 子域名 |

### 5.2 DNS 传播

- 添加 DNS 记录后等待 5-30 分钟全球传播
- 用 `dig geo.moy.com` 或 `nslookup` 验证

### 5.3 www 重定向

在所有域名上配置 `www` → 裸域的重定向：

- Nginx 方案：在 server block 中加入 301 重定向
- Vercel 方案：在 Domains 设置中启用 "Redirect www to root"

```
# Nginx www 重定向示例
server {
    server_name www.moy.com;
    return 301 https://moy.com$request_uri;
}
```

### 5.4 国内备案

如果使用国内服务器（阿里云/腾讯云），`.com` 域名需要 ICP 备案。备案期间网站不可访问，建议先用海外服务器或 Vercel 占位。

## 6. 上线顺序建议

| # | 站点 | 时机 | 原因 |
|:---:|---| --- | --- |
| 1 | `moy.com` | 最先 | 品牌官网是一切产品入口，先有一个在线的品牌门面 |
| 2 | `geo.moy.com` | 紧随其后 | GEO 销售页已有可收集线索的表单，可立即开始获客 |
| 3 | `api.moy.com` | 紧随其后 | API 产品页面可展示，建立开发者认知 |
| 4 | `app.moy.com` | 可延后 | MOY App 需要登录态和后端 API，依赖 §5 |
| 5 | `api.app.moy.com` | 第 4 步的同域 | 后端必须先于 app.moy.com 部署并验证可用 |
| 6 | `docs.moy.com` | S2+ | 文档中心是非紧急能力 |
| 7 | `status.moy.com` | S3+ | 状态页在有多站点的可用性监控需求后再建 |

### 6.1 第一阶段（1-3 天）

1. DNS 购买/配置 `moy.com`
2. 部署 `sites/official` → `moy.com`
3. 部署 `sites/geo` → `geo.moy.com`
4. 部署 `sites/api` → `api.moy.com`
5. 验证三站点可访问，SSL 正常

### 6.2 第二阶段（第 4-5 天）

6. 部署 `backend` → `api.app.moy.com`
7. 验证 Swagger (`api.app.moy.com/api/v1/docs`) 可访问
8. 部署 `frontend` → `app.moy.com`
9. 验证登录流程贯通

## 7. 安全注意事项

### 7.1 环境变量保护

| 规则 | 说明 |
| --- | --- |
| 前端只暴露 `VITE_` 前缀变量 | Vite 构建时注入，前端代码中可读取的只有这些 |
| 后端密钥不进版本控制 | `.env` 加入 `.gitignore`，生产环境通过平台面板或 `export` 设置 |
| 不提交 `.env` 到仓库 | 已删除所有历史中的 `.env` 文件 |

### 7.2 CORS 白名单

```typescript
// backend/src/main.ts
const ALLOWED_ORIGINS = [
  "https://moy.com",
  "https://app.moy.com",
  "https://geo.moy.com",
  "https://api.moy.com",
];

app.enableCors({
  origin: (origin, callback) => {
    if (!origin || ALLOWED_ORIGINS.includes(origin)) {
      callback(null, true);
    } else {
      callback(new Error("Not allowed by CORS"));
    }
  },
});
```

### 7.3 API Key 安全

| 规则 | 说明 |
| --- | --- |
| 不在前端代码中硬编码任何 API Key | 包括 OpenAI Key、GEO 后端 endpoint 中的 key |
| 前端表单调用的 endpoint 是公开的 | `VITE_GEO_LEAD_ENDPOINT` 本身不含密钥 |
| 后端代理上游 API | `backend` 持有供应商 Key（从环境变量读取），前端不感知 |

### 7.4 GEO 表单反垃圾

| 策略 | 说明 |
| --- | --- |
| honeypot 字段 | `_hint` 隐藏字段，机器人填写则静默丢弃 |
| IP 限流 | 同一 IP 每分钟最多 5 次 |
| 联系方式限频 | 同一联系方式 24 小时内最多 3 次 |

详见：[docs/GEO/08_§5](../docs/GEO/08_GEO线索收集后端设计.md#5-风控与反垃圾)。

### 7.5 防火墙与基础防护

```bash
# UFW 基础规则（方案 B）
sudo ufw allow 22       # SSH
sudo ufw allow 80       # HTTP
sudo ufw allow 443      # HTTPS
sudo ufw enable

# fail2ban 防暴力破解（方案 B）
sudo apt install fail2ban
sudo systemctl enable fail2ban
```

### 7.6 监控与告警（后期）

- `status.moy.com` — 公开服务状态页（S3+）
- Uptime 监控 — UptimeRobot（免费 50 条）或 Better Uptime
- 后端健康检查 — `GET /api/v1/health`
- 错误日志 — PM2 / Docker 日志 + 接入 Sentry / Grafana Loki（S3+）

## 8. CI/CD 建议（后期）

### 8.1 GitHub Actions 流程（方案 B）

```yaml
# .github/workflows/deploy-staging.yml（示例）
name: Deploy Staging

on:
  push:
    branches: [main]
    paths:
      - "sites/**"
      - "frontend/**"
      - "backend/**"

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 18 }

      - run: npm ci
        working-directory: sites/official
      - run: npm run build
        working-directory: sites/official
      - uses: appleboy/scp-action@v0
        with:
          host: ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          key: ${{ secrets.VPS_SSH_KEY }}
          source: "sites/official/dist/*"
          target: "/var/www/moy/official/"
```

### 8.2 Vercel 自动部署（方案 A）

Vercel 原生支持 git push → 自动构建部署，无需额外 CI 配置。每个 Vercel Project 关联对应目录即可。
