# MOY API 后续路线图

## 1. 文档用途

本文档定义 MOY API 从 MVP 到终局的阶段演进规划。每阶段有明确的目标、交付物和前置依赖。供产品规划和资源排期参考。

## 2. 阶段总览

```
S1 (MVP)          S2 (增强)         S3 (智能化)       S4 (企业级)       S5 (平台化)
───────          ───────          ───────          ───────          ───────
  转发             日志             路由              团队             统一
  认证             额度             fallback          企业             账号
  Key              限流             缓存              私有化           计费
  模型列表         统计             流式              账单             生态
  用量             风控             多模态            SLA
                                                                    与 MOY App
                                                                    MOY GEO 融合
```

## 3. S1：MVP（基础转发）

### 目标
用一个最小的技术闭环，验证"统一 API 网关 + 调用治理"的产品价值。

### 交付物

| 模块 | 内容 |
| --- | --- |
| 用户体系 | 邮箱注册/登录 |
| API Key | 创建、列表、启用/禁用、删除 |
| 模型列表 | `GET /v1/models` |
| Chat Completions | `POST /v1/chat/completions` 非流式转发 |
| 调用日志 | 记录每次调用，后台可查看 |
| 用量统计 | 概览面板 + 趋势图 + 按模型分组 |
| 额度 | 新用户赠送 ¥10，调用扣减，额度不足拦截 |
| 限流 | 按 Key QPS 限流 |
| 开发文档 | 快速开始、API Reference、代码示例 |
| 前端门户 | sites/api 的开发者后台 |

### 接入模型（S1）

| 供应商 | 模型 | 备注 |
| --- | --- | --- |
| OpenAI | gpt-4o, gpt-4o-mini | 通过官方 API |
| 月之暗面 | moonshot-v1-8k, moonshot-v1-32k | 通过官方 API |
| 智谱 | glm-4-flash | 通过官方 API |
| DeepSeek | deepseek-chat | 通过官方 API |

### 持续时间
预计 4-6 周。

## 4. S2：增强（日志、额度、限流、风控）

### 前置条件
- S1 上线，有真实开发者使用
- 收集到足够的用户反馈和使用数据

### 新增能力

| 能力 | 说明 |
| --- | --- |
| Stream 支持 | SSE 流式返回 `/v1/chat/completions`，支持 `stream: true` |
| Function Calling | 工具调用透传到上游 |
| 额度充值 | 微信/支付宝在线充值 |
| 高级限流 | 按 Project、按 IP、按模型的复合限流策略 |
| 日/周/月额度 | 设定周期性额度上限 |
| 日志高级查询 | 全文搜索、导出 CSV |
| 告警通知 | 额度不足邮件/站内通知 |
| 风险检测 | 异常调用模式检测、自动熔断 |
| 内容安全 | 接入内容安全审核服务 |
| 状态页 | api-status.moy.com 或 sites/api 内状态页组件自动化 |

### 新增模型（S2）

| 供应商 | 模型 |
| --- | --- |
| 阿里通义 | qwen-turbo, qwen-plus, qwen-max |
| 百度文心 | ernie-3.5, ernie-4.0 |
| Anthropic | claude-3-opus/sonnet/haiku（如合规允许） |
| Google | gemini-pro（如合规允许） |

### 持续时间
预计 6-8 周。

## 5. S3：智能化（模型路由、fallback、缓存）

### 前置条件
- S2 稳定运行
- 接入了 5+ 供应商，模型种类丰富
- 用量数据足以支撑智能路由决策

### 新增能力

| 能力 | 说明 |
| --- | --- |
| 智能模型路由 | 根据用户意图、预算、延迟要求自动选择最合适的模型 |
| 自动 Fallback | 主模型不可用时自动切换到备用模型 |
| 响应缓存 | 对相同/相似请求缓存结果，降低成本和延迟 |
| 语义缓存 | 对语义相似的请求复用缓存 |
| 多模态 | 支持图片、音频输入（上游支持即透传） |
| 竞价路由 | 多个供应商提供同类模型时，选最低价 |
| 完整 Billing | 发票系统、对账、消费明细导出 |
| 用量预测 | 基于历史数据预测下月用量和费用 |

### 新增模型（S3）

覆盖主流国产大模型和开源模型：

| 供应商 | 模型 |
| --- | --- |
| MiniMax | abab6.5 系列 |
| 零一万物 | yi-34b, yi-large |
| 百川 | baichuan3/4 |
| 字节豆包 | doubao-pro |

### 持续时间
预计 8-10 周。

## 6. S4：企业级（团队权限、企业套餐、私有化部署）

### 前置条件
- S3 稳定运行
- 有企业客户明确需求

### 新增能力

| 能力 | 说明 |
| --- | --- |
| 团队管理 | Project 内多成员、角色（Admin/Member/Viewer） |
| 企业套餐 | 月付/年付、阶梯价格、SLA 承诺 |
| 用量池 | 团队共享额度池，按成员分配 |
| 私有化部署 | 客户自有云部署方案 |
| 独立供应商账号 | 企业客户可使用自己的供应商 API Key |
| SSO | 支持 SAML/OIDC 企业单点登录 |
| 审计报告 | 合规审计报告自动生成 |
| 专属支持 | 企业客户专属技术支持通道 |

### 新增模型（S4）

| 供应商 | 说明 |
| --- | --- |
| 客户私有模型 | 客户自部署的开源模型接入 |
| Azure OpenAI | 企业合规通道 |

### 持续时间
预计 10-12 周。

## 7. S5：平台化（统一账号、计费融合、生态）

### 前置条件
- S4 稳定运行
- MOY App 和 MOY GEO 均有成熟用户基础

### 新增能力

| 能力 | 说明 |
| --- | --- |
| 统一账号 | MOY App / MOY GEO / MOY API 共享一套用户体系 |
| 统一计费 | 三产品共享余额/积分/套餐 |
| 统一后台 | 一个后台管理所有 MOY 产品 |
| OpenAPI 生态 | 开放更多 API，支持第三方集成 |
| 开发者社区 | 文档、论坛、Showcase |
| 合作伙伴计划 | 代理商、技术合作伙伴体系 |

### 新增模型（S5）

- 持续接入主流供应商的新模型
- 支持客户自定义模型接入

### 持续时间
持续迭代。

## 9. 每个阶段的度量指标

| 阶段 | 关键指标 |
| --- | --- |
| S1 | 注册开发者数、API 调用量、模型使用分布、额度消耗速度 |
| S2 | 付费转化率、充值金额、stream 使用占比、风控拦截数 |
| S3 | 缓存命中率、fallback 触发次数、路由质量分、多模态调用量 |
| S4 | 企业客户数、私有化部署数、ARR/MRR |
| S5 | 统一账号激活率、跨产品渗透率、生态伙伴数 |

## 10. 风险与假设

| 风险 | 影响 | 缓解方案 |
| --- | --- | --- |
| 供应商 API 涨价或停服 | 成本上升、模型不可用 | 多供应商策略，不绑定单一供应商 |
| 监管政策变化 | 某些模型或供应商不可用 | 密切关注监管动态，国产模型优先 |
| AI 市场降温 | 开发者需求下降 | 差异化在治理层，不完全依赖 AI 热度 |
| 大厂竞争 | 阿里云/腾讯云推出类似产品 | 聚焦中小开发者和垂直场景 |
| 滥用导致供应商封号 | 影响所有开发者 | 严格风控，快速响应 |
