# MOY API 调用日志与成本统计

## 1. 文档用途

本文档定义 MOY API 的调用日志记录规范、成本计算模型、统计方案。供后端开发实现日志中间件和统计接口参考。

## 2. 调用日志表

表结构在 [02_API_Key与用量模型](./02_API_Key与用量模型.md) 中已定义，本章重点说明各字段的记录规则和业务含义。

### 2.1 字段记录规则

| 字段 | 记录时机 | 来源 |
| --- | --- | --- |
| `id` | 创建日志记录时 | 系统生成 UUID |
| `project_id` | 鉴权中间件注入 | 从 api_key 反查 |
| `api_key_id` | 鉴权中间件注入 | api_key.id |
| `user_id` | 鉴权中间件注入 | api_user.id |
| `model` | 请求参数 | `req.body.model` |
| `provider` | 模型配置映射 | 系统内部，如 `openai`、`zhipu` |
| `request_id` | 请求头或系统生成 | `X-MOY-Request-Id` header，无则生成 UUID |
| `prompt_tokens` | 上游响应返回后 | `response.usage.prompt_tokens` |
| `completion_tokens` | 上游响应返回后 | `response.usage.completion_tokens` |
| `total_tokens` | 上游响应返回后 | 同上，或 prompt+completion |
| `cost_amount` | 上游响应返回后 | 根据 tokens * 定价计算 |
| `status` | 流程结束 | `success` / `error` / `rate_limited` / `quota_exceeded` |
| `error_code` | 出错时 | 来自供应商或内部产生的错误码 |
| `error_message` | 出错时 | 错误描述（不包含敏感信息） |
| `latency_ms` | 上游响应返回后 | 从发出请求到收到响应的时间差 |
| `ip_address` | 请求进入时 | `req.ip` |
| `user_agent` | 请求进入时 | `req.headers['user-agent']` |
| `created_at` | 创建日志记录时 | 请求进入时间 |

### 2.2 status 枚举

| 值 | 说明 | 是否扣费 |
| --- | --- | --- |
| `success` | 调用成功，正常返回 | 是 |
| `error` | 调用失败（含参数错误、上游错误） | 视错误类型 |
| `rate_limited` | 触发限流，未转发到上游 | 否 |
| `quota_exceeded` | 额度不足，未转发到上游 | 否 |

### 2.3 隐私与合规

- **不记录 message content**：不存储用户发送的消息内容和 AI 返回的内容
- **不记录个人可识别信息（PII）**：若从 IP 等字段可推演，确保日志存储加密
- **日志保留期**：原始日志保留 90 天，聚合统计数据保留 3 年
- **日志访问权限**：仅运维和管理员可查看原始日志

## 3. 成本计算模型

### 3.1 定价表

每个模型的成本由 prompt 和 completion 两部分组成，按千 token 计价（人民币）：

| model | provider | prompt/1K | completion/1K | 货币 |
| --- | --- | --- | --- | --- |
| `gpt-4o` | openai | ¥0.015 | ¥0.060 | CNY |
| `gpt-4o-mini` | openai | ¥0.001 | ¥0.004 | CNY |
| `moonshot-v1-8k` | moonshot | ¥0.012 | ¥0.012 | CNY |
| `moonshot-v1-32k` | moonshot | ¥0.024 | ¥0.024 | CNY |
| `glm-4-flash` | zhipu | ¥0.000 | ¥0.000 | CNY |
| `deepseek-chat` | deepseek | ¥0.010 | ¥0.020 | CNY |

实际定价以运行时的模型配置为准，以上为参考值。

### 3.2 成本计算公式

```
cost = (prompt_tokens / 1000) * prompt_price + (completion_tokens / 1000) * completion_price
```

结果保留 8 位小数。

### 3.3 成本与售价

MOY API 的盈利模型有两种模式（MVP 阶段按模式 A）：

| 模式 | 说明 | 适用阶段 |
| --- | --- | --- |
| A. 成本定价 | 直接按上游成本计费，盈利来自后续增值服务 | MVP |
| B. 加价定价 | 在成本上浮一定比例（如 15-30%） | S3+ |

MVP 阶段采用模式 A，让早期开发者以接近成本的价格使用，降低获客门槛。S3 阶段根据业务数据决定是否切换到模式 B。

### 3.4 成本记录示例

一次 GPT-4o 调用：
```
prompt_tokens: 500
completion_tokens: 200

cost = (500/1000)*0.015 + (200/1000)*0.06
     = 0.0075 + 0.012
     = ¥0.0195
```

## 4. 日志记录时机与事务处理

### 4.1 成功路径

```
请求进入 → 创建 UsageLog(status='pending') → 鉴权通过 → 限流通过 →
参数校验通过 → 额度检查通过 → 转发上游 → 收到响应 → 填充 tokens/cost →
UPDATE status='success', tokens, cost, latency_ms
```

### 4.2 失败路径——上游错误

```
请求进入 → ... → 转发上游 → 上游返回 5xx → 
UPDATE status='error', error_code='upstream_xxx', error_message='...'
→ 不扣费
```

### 4.3 失败路径——参数错误

```
请求进入 → 参数校验失败 →
INSERT status='error', error_code='invalid_request'
→ 不扣费
```

### 4.4 失败路径——上游返回错误但已消耗 token

部分供应商在返回错误时仍会消耗 token（如内容安全拦截）。此时：

| 情况 | 处理 |
| --- | --- |
| 供应商在 error response 中包含了 usage | 按实际 usage 记录并扣费 |
| 供应商未返回 usage | 不扣费，记录 error_code 和 error_message |

## 5. 统计查询接口

### 5.1 概览统计

```
GET /api/dashboard/stats?period=today|this_month|total
```

返回：

```json
{
  "period": "this_month",
  "total_calls": 1250,
  "total_tokens": 450000,
  "total_cost": 12.50,
  "success_rate": 98.5,
  "avg_latency_ms": 850
}
```

### 5.2 按模型统计

```
GET /api/dashboard/stats/by-model?period=this_month
```

返回：

```json
{
  "period": "this_month",
  "models": [
    {
      "model": "gpt-4o",
      "calls": 500,
      "tokens": 200000,
      "cost": 8.50
    },
    {
      "model": "gpt-4o-mini",
      "calls": 750,
      "tokens": 250000,
      "cost": 4.00
    }
  ]
}
```

### 5.3 按天趋势

```
GET /api/dashboard/stats/daily?days=30
```

返回：

```json
{
  "days": 30,
  "data": [
    {
      "date": "2026-05-01",
      "calls": 45,
      "tokens": 15000,
      "cost": 0.45
    },
    {
      "date": "2026-05-02",
      "calls": 52,
      "tokens": 18000,
      "cost": 0.52
    }
  ]
}
```

### 5.4 调用日志列表

```
GET /api/logs?page=1&page_size=20&model=gpt-4o&status=success&start_date=2026-05-01&end_date=2026-05-31
```

返回分页的日志列表，仅包含非敏感字段。

## 6. 日志数据清理策略

| 数据 | 保留策略 | 处理方式 |
| --- | --- | --- |
| 原始 usage_log | 90 天 | 定时任务归档到冷存储后删除 |
| 按天聚合统计 | 3 年 | 保留在主库 |
| 按月聚合统计 | 永久 | 保留在主库 |
| API Key 哈希 | 永久（直到 Key 被删除） | 正常保留 |

## 7. 与外部系统集成

### 7.1 将来接入监控系统

日志数据在 MVP 后期或 S2 接入 Prometheus metrics：

```
# HELP moy_api_calls_total Total API calls
# TYPE moy_api_calls_total counter
moy_api_calls_total{model="gpt-4o",status="success"} 500
moy_api_calls_total{model="gpt-4o",status="error"} 3

# HELP moy_api_tokens_total Total tokens consumed
# TYPE moy_api_tokens_total counter
moy_api_tokens_total{model="gpt-4o",type="prompt"} 100000
moy_api_tokens_total{model="gpt-4o",type="completion"} 80000

# HELP moy_api_cost_total Total cost in CNY
# TYPE moy_api_cost_total counter
moy_api_cost_total{model="gpt-4o"} 8.50

# HELP moy_api_latency_ms API latency in milliseconds
# TYPE moy_api_latency_ms histogram
moy_api_latency_ms_bucket{model="gpt-4o",le="100"} 50
moy_api_latency_ms_bucket{model="gpt-4o",le="500"} 350
moy_api_latency_ms_bucket{model="gpt-4o",le="1000"} 480
```

### 7.2 将来接入告警

- 错误率 > 5% 持续 5 分钟 → 告警
- 上游供应商全部不可用 → 紧急告警
- 单用户调用量突增 300% → 风控告警
