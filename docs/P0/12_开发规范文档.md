## 一、文档目的

本文档作为 MOY 项目首期 MVP 的**研发执行规范基线**，用于：

1. 建立统一的代码结构、命名、分支、提交标准
2. 规范环境配置、联调、测试与变更管理流程
3. 明确 AI 编程工具（Trae）的使用边界与审查要求
4. 保证多人协作时代码可维护、可追溯、可评审
5. 确保开发活动不突破 PRD 定义的产品边界

**阅读建议：**
- 技术负责人：全文阅读，负责规范落地与监督
- 前端/后端开发：重点阅读目录规范、命名规范、开发规范
- AI 工程师：重点阅读 AI 模块开发规范、AI 编程工具使用规范
- 测试工程师：重点阅读测试与验收协作规范
- 所有人员：必须阅读 AI 编程工具使用规范

**重要说明：** 本文档具有约束力，所有开发活动必须遵循本规范。违反规范的代码不予合并。

## 四、技术协作原则

### 4.1 核心协作原则

| 原则 | 说明 | 违反后果 |
|------|------|----------|
| **先文档后开发** | 任何功能开发前，必须确认 PRD、HLD、API 文档已定义清晰 | 代码不予合并 |
| **先接口后联调** | 前后端联调前，必须完成 API 文档评审和 Mock 数据准备 | 联调不予开始 |
| **先主流程后边缘场景** | 优先完成核心业务流程，边缘场景延后处理 | 评审不予通过 |
| **先 P0 后 P1** | 严格按照 RTM 定义的优先级顺序开发 | 任务不予验收 |
| **不突破 PRD 边界** | 任何实现不得超出 PRD 定义的功能范围 | 代码必须回滚 |

### 4.2 开发活动边界

| 活动类型 | 允许范围 | 禁止范围 |
|----------|----------|----------|
| 功能开发 | PRD 定义的 P0/P1 功能 | PRD 未定义的功能 |
| 技术选型 | HLD 定义的技术栈 | 未经审批的新技术引入 |
| 数据结构 | DBD 定义的表结构 | 擅自新增/修改表结构 |
| 接口设计 | API 文档定义的接口 | 擅自新增/修改接口 |
| AI 能力 | HLD 定义的 AI 场景 | 擅自扩展 AI 应用范围 |

## 六、命名规范

### 6.1 文件命名规范

| 文件类型 | 命名规范 | 示例 |
|----------|----------|------|
| React 组件 | PascalCase | `CustomerForm.tsx` |
| React 组件目录 | kebab-case | `customer-form/` |
| Hook 文件 | camelCase + use 前缀 | `useCustomer.ts` |
| Service 文件 | camelCase + .service 后缀 | `customer.service.ts` |
| Controller 文件 | camelCase + .controller 后缀 | `customer.controller.ts` |
| DTO 文件 | camelCase + .dto 后缀 | `create-customer.dto.ts` |
| Entity 文件 | camelCase + .entity 后缀 | `customer.entity.ts` |
| 类型文件 | camelCase + .types 后缀 | `customer.types.ts` |
| 常量文件 | camelCase + .constants 后缀 | `status.constants.ts` |
| 工具文件 | camelCase | `format.ts` |
| 测试文件 | 源文件名 + .spec/.test | `customer.service.spec.ts` |
| 样式文件 | 与组件同名 | `CustomerForm.module.css` |
| 配置文件 | kebab-case | `next.config.js` |

### 6.2 变量命名规范

| 变量类型 | 命名规范 | 示例 |
|----------|----------|------|
| 普通变量 | camelCase | `customerName` |
| 常量 | UPPER_SNAKE_CASE | `MAX_PAGE_SIZE` |
| 枚举值 | UPPER_SNAKE_CASE | `CustomerStatus.ACTIVE` |
| 私有变量 | _camelCase | `_internalState` |
| 布尔变量 | is/has/can 前缀 | `isActive`, `hasPermission` |
| 数组变量 | 复数形式 | `customers`, `items` |
| Map/Set 变量 | 描述性名称 + Map/Set | `customerMap`, `idSet` |
| 回调函数 | on 前缀 | `onSubmit`, `onClick` |
| 异步变量 | async 后缀或无特殊 | `fetchCustomers` |

### 6.3 接口命名规范

| 接口类型 | 命名规范 | 示例 |
|----------|----------|------|
| RESTful 资源 | 复数名词 | `/api/v1/customers` |
| 子资源 | 父资源/{id}/子资源 | `/api/v1/customers/{id}/contacts` |
| 操作接口 | 动作名词 | `/api/v1/leads/{id}/convert` |
| 查询参数 | camelCase | `?pageSize=20&sortBy=name` |
| 路径参数 | camelCase | `/{customerId}` |
| 请求体字段 | camelCase | `{ "customerName": "xxx" }` |
| 响应体字段 | camelCase | `{ "totalCount": 100 }` |

### 6.4 数据表命名规范

| 表类型 | 命名规范 | 示例 |
|----------|----------|------|
| 业务表 | 复数名词，snake_case | `customers` |
| 关联表 | 表1_表2 | `customer_tags` |
| 系统表 | sys_ 前缀 | `sys_audit_logs` |
| 字段名 | snake_case | `customer_name` |
| 主键 | id | `id` |
| 外键 | 关联表_singular_id | `customer_id` |
| 索引 | idx_表名_字段 | `idx_customers_status` |
| 唯一约束 | uk_表名_字段 | `uk_users_username` |

### 6.5 枚举命名规范

| 枚举类型 | 命名规范 | 示例 |
|----------|----------|------|
| 枚举类型名 | PascalCase | `CustomerStatus` |
| 枚举值 | UPPER_SNAKE_CASE | `ACTIVE`, `INACTIVE` |
| TypeScript 枚举 | `export enum` | 见下方示例 |

**TypeScript 枚举示例：**

```typescript

  INACTIVE = 'inactive',

export enum LeadStatus {

  FOLLOWING = 'following',

  INVALID = 'invalid'

```

```typescript
// common/decorators/audit-log.decorator.ts
export const AuditLog = (action: string, resourceType: string) => {
  return SetMetadata('AUDIT_LOG', { action, resourceType });
};

// customer.service.ts
@Injectable()
export class CustomerService {
  @AuditLog('create', 'customer')
  async create(dto: CreateCustomerDto, userId: number) {
    const customer = await this.repository.save(dto);
    return customer;
  }

  @AuditLog('update', 'customer')
```

```typescript
    const results = await Promise.race([
```

```typescript
      return {

        suggestions: await this.generateSuggestions(question),
        auto_send: false,
        require_confirmation: true
```

```typescript
export function maskPhone(phone: string): string {
  if (!phone || phone.length !== 11) return phone;
  return phone.replace(/(\d{3})\d{4}(\d{4})/, '$1****$2');
}

export function maskEmail(email: string): string {
  if (!email) return email;
  const [local, domain] = email.split('@');
  const maskedLocal = local.length > 3 
    ? local.slice(0, 3) + '****' 
    : '****';
  return `${maskedLocal}@${domain}`;
}
```
