## 一、文档目的

本文档作为 MOY 项目 P0 阶段文档体系的**治理基线**，定义：

1. 文档生成原则与边界
2. 命名、版本、引用规范
3. 变更记录规则
4. 信息真实性约束
5. AI 协作代理行为边界

所有 P0 阶段文档的生成、评审、迭代必须遵循本规则。

## 二、文档生成原则

### 2.1 核心原则

| 编号 | 原则 | 说明 |
|------|------|------|
| GP-01 | 单一事实源 | 同一业务定义只在一个主文档拥有最终解释权，禁止多处定义冲突 |
| GP-02 | 先文档后开发 | 核心链路、边界、验收标准未确定前，不进入大规模编码 |
| GP-03 | 版本化管理 | 所有文档必须记录版本号、变更人、日期、变更摘要、审批状态 |
| GP-04 | 可追溯性 | 业务需求需可追溯到 PRD、设计、开发、测试、上线验证 |
| GP-05 | 显式假设 | 未知信息必须显式标注，不得以确定性语气表述未证实内容 |
| GP-06 | 阶段化产出 | 按阶段迭代，不追求一次性完成全部文档 |
| GP-07 | 证据支撑 | 关键结论必须有来源登记，来源类型需明确标注 |
| GP-08 | 保守表达 | 涉及法规、合规、财务估算，必须保守表达并注明假设前提 |

### 2.2 禁止事项

| 编号 | 禁止行为 | 理由 |
|------|----------|------|
| FB-01 | 禁止臆造市场数据 | 市场规模、增长率、竞品数据必须有来源或明确标注为假设 |
| FB-02 | 禁止臆造法规结论 | 合规内容仅做风险识别，不得给出法律绝对结论 |
| FB-03 | 禁止臆造客户访谈结论 | 无真实访谈数据时，必须标注"模拟结论禁止作为正式事实" |
| FB-04 | 禁止编造财务数据 | 客单价、转化率、ROI 等必须写明估算方法与前提条件 |
| FB-05 | 禁止脱离现实范围 | 必须区分"长期总方向"与"首期 MVP 落地边界" |

## 三、信息标注规范

### 3.1 未确认信息标注

所有无法确认的信息必须使用以下标注：

| 标注类型 | 标签 | 使用场景 | 示例 |
|----------|------|----------|------|
| 待确认 | `[TBD]` | 信息需要人工确认或补充 | 目标用户规模：[TBD] 需业务方确认 |
| 假设 | `[Assumption]` | 基于合理推断但未验证 | [Assumption] 假设首年目标客户 50 家 |
| 待验证 | `[Pending Validation]` | 需要通过调研或实验验证 | [Pending Validation] 需通过用户访谈验证 |
| 模拟数据 | `[Simulated]` | 仅为示例，禁止作为正式事实 | [Simulated] 模拟访谈结论，禁止作为正式事实 |
| 估算 | `[Estimated]` | 基于方法论的估算，需注明前提 | [Estimated] 基于 X 方法估算，前提条件为 Y |

### 3.2 来源类型标注

| 来源类型 | 标签 | 可信度 | 说明 |
|----------|------|--------|------|
| 事实 | `[Fact]` | 高 | 来自权威来源的可验证数据 |
| 推断 | `[Inference]` | 中 | 基于事实的逻辑推演 |
| 假设 | `[Hypothesis]` | 低 | 未经证实的合理猜测 |
| 外部来源 | `[External:来源]` | 视来源而定 | 来自外部文献、报告、网站 |

## 五、版本规范

### 5.1 版本号规则

| 版本阶段 | 版本号范围 | 说明 |
|----------|------------|------|
| 探索版 | 0.x | 草案阶段，内容可能大幅调整 |
| 正式基线 | 1.0 | 首个正式发布版本，经评审确认 |
| 次版本更新 | 1.x | 局部补充、修正，无结构性变化 |
| 主版本更新 | x.0 | 结构性重大变化，需重新评审 |

### 5.2 版本记录格式

每个文档末尾必须包含版本记录表：

```markdown
## 版本记录

| 版本 | 日期 | 作者 | 变更摘要 | 状态 |
|------|------|------|----------|------|
| v0.1 | 2026-04-04 | XXX | 初稿 | 草案 |
```

## 六、引用规范

### 6.1 文档间引用

文档间引用必须使用以下格式：

```markdown
详见 [文档名称](文件路径) 章节 X.X
```

**示例：**
```markdown
业务背景详见 [项目介绍](./02_项目介绍.md) 章节 2.1
功能需求详见 [PRD](./06_PRD_产品需求规格说明书_v0.1.md) 章节 4.0
```

### 6.2 外部来源引用

外部来源引用必须包含：

```markdown
[来源类型] 来源名称，发布机构/作者，发布日期，URL（如有）
```

**示例：**
```markdown
[Fact] 2024年中国SaaS行业研究报告，艾瑞咨询，2024-03，https://...
[External:Gartner] Gartner Magic Quadrant for CRM，Gartner，2024
```

## 七、变更记录规则

### 7.1 变更类型

| 类型 | 标识 | 影响范围 | 处理要求 |
|------|------|----------|----------|
| 新增 | ADD | 新增章节或内容 | 记录新增位置与原因 |
| 修改 | MOD | 修改现有内容 | 记录修改前后对比 |
| 删除 | DEL | 删除内容 | 记录删除原因与备份位置 |
| 状态变更 | STATUS | 文档状态变化 | 记录变更前后的状态 |

### 7.2 变更记录位置

1. **文档末尾版本记录表**：记录每次版本变更
2. **变更摘要章节**：重大变更需在文档开头增加变更摘要
3. **RTM 同步更新**：需求变更必须同步更新 RTM

### 7.3 变更影响链

```
PRD 变更 -> 同步更新 BRD -> 同步更新 RTM -> 评估 HLD/API/测试用例影响
```

## 八、AI 协作代理行为边界

### 8.1 AI 代理角色定位

AI 协作代理定位为**文档工程师**，职责边界：

| 允许行为 | 禁止行为 |
|----------|----------|
| 按规范生成文档骨架 | 擅自改变产品方向 |
| 整理已有信息 | 臆造未提供的数据 |
| 标注待确认事项 | 以确定语气表述假设 |
| 提供结构化建议 | 替代人工决策 |
| 执行格式规范 | 跳过评审直接入库 |

### 8.2 AI 生成文档必读清单

生成任何文档前，AI 代理必须先读取：

1. 本治理规则文档（00_AGENTS.md）
2. 来源登记表（01_Source_Register.md）
3. 最新项目介绍（02_项目介绍.md）
4. 最新 BRD（05_BRD_业务需求说明书.md）
5. 最新 PRD（06_PRD_产品需求规格说明书_v0.1.md）
6. 最新 RTM（07_RTM_需求跟踪矩阵.md）

### 8.3 AI 输出规范

| 规范项 | 要求 |
|--------|------|
| 语言 | 默认中文输出 |
| 骨架 | 目的 -> 范围 -> 输入 -> 正文 -> 验收 -> 待确认项 |
| 假设标注 | 未确认信息必须显式标注 |
| 保守表达 | 安全、隐私、合同、合规、财务估算必须写明假设前提 |
| 变更同步 | 文档改动必须写变更摘要并同步更新 RTM |

## 九、文档状态定义

| 状态 | 英文 | 说明 | 后续动作 |
|------|------|------|----------|
| 草案 | Draft | 初稿，内容可能大幅调整 | 内部评审 |
| 评审中 | Under Review | 已提交评审，等待反馈 | 收集意见、修订 |
| 已确认 | Approved | 评审通过，作为正式基线 | 可进入下一阶段 |
| 已发布 | Released | 正式发布，对外可见 | 按变更流程维护 |
| 已归档 | Archived | 历史版本，仅供参考 | 不再主动维护 |

## 十、防飘移机制

### 10.1 生成前检查

每次生成新文档前必须检查：

- [ ] 是否已指定上游输入文档
- [ ] 上游输入文档是否为最新版本
- [ ] 是否存在冲突定义
- [ ] 是否需要标注待确认事项

### 10.2 评审重点

文档评审重点审查：

1. **战略一致性**：是否与项目定位、目标一致
2. **范围边界**：是否超出 MVP 边界或遗漏核心功能
3. **指标可验证性**：指标是否有明确定义与计算口径
4. **来源可靠性**：关键结论是否有可靠来源支撑
5. **假设合理性**：假设是否合理、是否需要验证

### 10.3 数值结论规范

所有数值结论必须包含：

1. **计算口径**：数值如何得出
2. **假设前提**：基于什么条件
3. **敏感因素**：哪些因素会影响结果
4. **场景区分**：保守/中性/乐观场景
