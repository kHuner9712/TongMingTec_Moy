## 二、文档层级与基线关系

### 2.1 文档层级定义

本项目文档分为三个层级：

| 层级 | 名称 | 定义 | 变更影响 |
|------|------|------|----------|
| **一级基线文档** | Master Document | 业务定义的权威来源，变更需评审 | 变更必须同步更新所有下游文档 |
| **二级从属文档** | Derived Document | 依赖一级基线文档生成，不得与基线冲突 | 需随基线变更同步更新 |
| **三级支撑文档** | Supporting Document | 辅助说明、记录类文档 | 独立维护 |

## 四、范围治理规则

### 4.1 范围定义统一

本项目采用以下范围术语，**禁止混用**：

| 术语 | 定义 | 包含内容 |
|------|------|----------|
| **首期完整版** | 第一个可交付的生产版本 | P0全部功能 + P1全部功能 |
| **P0** | 核心业务功能 | 客户管理、线索管理、会话管理、商机管理、工单管理 |
| **P1** | 增强业务功能 | 知识库管理、数据看板 |
| **P2** | 后续迭代功能 | 高级分析、扩展集成等（不在首期范围） |

### 4.2 首期完整版范围清单

#### P0 核心功能（必须完成）

| 模块 | 功能点 | 优先级 | 说明 |
|------|--------|--------|------|
| 客户管理 | 客户档案、客户列表、客户详情、客户标签 | P0 | 核心数据实体 |
| 线索管理 | 线索录入、线索分配、线索跟进、线索转化 | P0 | 销售获客入口 |
| 会话管理 | 多渠道接入、会话列表、消息收发、智能回复 | P0 | 客服核心场景 |
| 商机管理 | 商机创建、阶段推进、商机预测、商机分析 | P0 | 销售核心场景 |
| 工单管理 | 工单创建、工单流转、SLA管理、工单统计 | P0 | 售后服务场景 |

#### P1 增强功能（必须完成）

| 模块 | 功能点 | 优先级 | 说明 |
|------|--------|--------|------|
| 知识库 | 知识分类、知识条目、知识检索、AI问答 | P1 | AI能力支撑 |
| 数据看板 | 销售看板、客服看板、运营看板 | P1 | 数据分析支撑 |

#### P2 后续功能（不在首期范围）

| 模块 | 功能点 | 优先级 | 说明 |
|------|--------|--------|------|
| 高级分析 | 客户画像分析、销售预测模型 | P2 | 后续迭代 |
| 扩展集成 | CRM集成、呼叫中心集成 | P2 | 后续迭代 |
| 移动端 | 移动App、小程序 | P2 | 后续迭代 |

### 4.3 范围表述规范

| 场景 | 正确表述 | 错误表述 |
|------|----------|----------|
| 定义首期范围 | "首期完整版包含P0和P1全部功能" | "MVP包含P0功能" |
| 定义核心功能 | "P0为核心业务功能" | "P0为MVP功能" |
| 定义增强功能 | "P1为增强业务功能，属于首期完整版" | "P1为后续迭代功能" |
| 定义后续功能 | "P2为后续迭代功能，不在首期范围" | "P2为二期功能" |

### 4.4 范围变更控制

范围变更必须遵循以下流程：

1. **提出变更**：在 15_未决事项清单.md 中记录
2. **影响评估**：评估对 BRD/PRD/RTM/HLD/API/DBD 的影响
3. **评审决策**：经项目组评审确认
4. **同步更新**：更新所有受影响的文档
5. **版本升级**：受影响文档版本号升级

## 六、文档生成原则

### 6.1 核心原则

| 编号 | 原则 | 说明 |
|------|------|------|
| GP-01 | 单一事实源 | 同一业务定义只在一个主文档拥有最终解释权，禁止多处定义冲突 |
| GP-02 | 先文档后开发 | 核心链路、边界、验收标准未确定前，不进入大规模编码 |
| GP-03 | 版本化管理 | 所有文档必须记录版本号、变更人、日期、变更摘要、审批状态 |
| GP-04 | 可追溯性 | 业务需求需可追溯到 PRD、设计、开发、测试、上线验证 |
| GP-05 | 显式假设 | 未知信息必须显式标注，不得以确定性语气表述未证实内容 |
| GP-06 | 阶段化产出 | 按阶段迭代，不追求一次性完成全部文档 |
| GP-07 | 证据支撑 | 关键结论必须有来源登记，来源类型需明确标注 |
| GP-08 | 保守表达 | 涉及法规、合规、财务估算，必须保守表达并注明假设前提 |

### 6.2 禁止事项

| 编号 | 禁止行为 | 理由 |
|------|----------|------|
| FB-01 | 禁止臆造市场数据 | 市场规模、增长率、竞品数据必须有来源或明确标注为假设 |
| FB-02 | 禁止臆造法规结论 | 合规内容仅做风险识别，不得给出法律绝对结论 |
| FB-03 | 禁止臆造客户访谈结论 | 无真实访谈数据时，必须标注"模拟结论禁止作为正式事实" |
| FB-04 | 禁止编造财务数据 | 客单价、转化率、ROI 等必须写明估算方法与前提条件 |
| FB-05 | 禁止脱离现实范围 | 必须区分"长期总方向"与"首期完整版落地边界" |

## 七、信息标注规范

### 7.1 未确认信息标注

所有无法确认的信息必须使用以下标注：

| 标注类型 | 标签 | 使用场景 | 示例 |
|----------|------|----------|------|
| 待确认 | `[TBD]` | 信息需要人工确认或补充 | 目标用户规模：[TBD] 需业务方确认 |
| 假设 | `[Assumption]` | 基于合理推断但未验证 | [Assumption] 假设首年目标客户 50 家 |
| 待验证 | `[Pending Validation]` | 需要通过调研或实验验证 | [Pending Validation] 需通过用户访谈验证 |
| 模拟数据 | `[Simulated]` | 仅为示例，禁止作为正式事实 | [Simulated] 模拟访谈结论，禁止作为正式事实 |
| 估算 | `[Estimated]` | 基于方法论的估算，需注明前提 | [Estimated] 基于 X 方法估算，前提条件为 Y |

### 7.2 来源类型标注

| 来源类型 | 标签 | 可信度 | 说明 |
|----------|------|--------|------|
| 事实 | `[Fact]` | 高 | 来自权威来源的可验证数据 |
| 推断 | `[Inference]` | 中 | 基于事实的逻辑推演 |
| 假设 | `[Hypothesis]` | 低 | 未经证实的合理猜测 |
| 外部来源 | `[External:来源]` | 视来源而定 | 来自外部文献、报告、网站 |

### 8.1 文档命名规则

```

详见 [文档名称](文件路径) 章节 X.X
```

**示例：**
```markdown
业务背景详见 [项目介绍](./02_项目介绍.md) 章节 2.1
功能需求详见 [PRD](./06_PRD_产品需求规格说明书.md) 章节 4.0
```

### 9.2 外部来源引用

外部来源引用必须包含：

```markdown
[来源类型] 来源名称，发布机构/作者，发布日期，URL（如有）
```

**示例：**
```markdown
[Fact] 2024年中国SaaS行业研究报告，艾瑞咨询，2024-03，https://...
[External:Gartner] Gartner Magic Quadrant for CRM，Gartner，2024
```

## 十、变更记录规则

### 10.1 变更类型

| 类型 | 标识 | 影响范围 | 处理要求 |
|------|------|----------|----------|
| 新增 | ADD | 新增章节或内容 | 记录新增位置与原因 |
| 修改 | MOD | 修改现有内容 | 记录修改前后对比 |
| 删除 | DEL | 删除内容 | 记录删除原因与备份位置 |
| 状态变更 | STATUS | 文档状态变化 | 记录变更前后的状态 |

### 10.2 变更记录位置

1. **文档末尾版本记录表**：记录每次版本变更
2. **变更摘要章节**：重大变更需在文档开头增加变更摘要
3. **RTM 同步更新**：需求变更必须同步更新 RTM

### 10.3 变更影响链

```
PRD 变更 -> 同步更新 BRD -> 同步更新 RTM -> 评估 HLD/API/DBD 影响 -> 同步更新
```

## 十二、防飘移机制

### 12.1 生成前检查

每次生成新文档前必须检查：

- [ ] 是否已指定上游输入文档
- [ ] 上游输入文档是否为最新版本
- [ ] 是否存在冲突定义
- [ ] 是否需要标注待确认事项

### 12.2 评审重点

文档评审重点审查：

1. **战略一致性**：是否与项目定位、目标一致
2. **范围边界**：是否超出首期完整版边界或遗漏核心功能
3. **指标可验证性**：指标是否有明确定义与计算口径
4. **来源可靠性**：关键结论是否有可靠来源支撑
5. **假设合理性**：假设是否合理、是否需要验证

### 12.3 数值结论规范

所有数值结论必须包含：

1. **计算口径**：数值如何得出
2. **假设前提**：基于什么条件
3. **敏感因素**：哪些因素会影响结果
4. **场景区分**：保守/中性/乐观场景
