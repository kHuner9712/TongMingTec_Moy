## 3. 环境变量清单（DEV/TEST/PROD）

### 3.1 通用变量
| 变量名 | DEV | TEST | PROD | 说明 |
| --- | --- | --- | --- | --- |
| NODE_ENV | development | test | production | 运行环境 |
| APP_PORT | 必填 | 必填 | 必填 | API 端口 |
| WS_PORT | 必填 | 必填 | 必填 | WS 端口 |
| WEB_BASE_URL | 必填 | 必填 | 必填 | 前端地址 |
| API_BASE_URL | 必填 | 必填 | 必填 | API 地址 |
| JWT_SECRET | 必填 | 必填 | 必填（密钥管理） | JWT 签名密钥 |
| JWT_EXPIRES_IN | 可选 | 可选 | 可选 | access token 过期时间 |
| JWT_REFRESH_EXPIRES_IN | 可选 | 可选 | 可选 | refresh token 过期时间 |
| REQUEST_TIMEOUT_MS | 可选 | 可选 | 可选 | API 超时 |

## 4. 密钥管理约束
- `JWT_SECRET`、`DB_PASSWORD`、`REDIS_PASSWORD`、`AI_API_KEY` 必须从密钥系统注入。
- 禁止写入 `.env.example` 的真实值。
- 日志中禁止输出密钥明文。

## 5. 部署顺序（服务级）
1. PostgreSQL（含备份验证）
2. Redis
3. API 服务
4. WS Gateway
5. Web 前端
6. 网关路由切换

## 6. 数据库迁移执行顺序
- 执行顺序以 `10_DBD_数据模型与数据字典.md` 第 8 章为准。
- 执行策略：
  - `up`：严格顺序执行
  - `down`：逆序回滚
- 每次发布必须记录：migration 版本、执行人、开始/结束时间、结果。

## 8. 告警项（P0）
| 告警项 | 阈值 | 级别 |
| --- | --- | --- |
| API 5xx 比例 | >1% 持续 5 分钟 | P1 |
| API P95 延迟 | >800ms 持续 10 分钟 | P2 |
| WS 心跳超时率 | >5% 持续 5 分钟 | P1 |
| AI 超时率 | >10% 持续 10 分钟 | P2 |
| DB 连接池占用 | >85% 持续 5 分钟 | P1 |
| Redis 内存占用 | >80% 持续 10 分钟 | P2 |

## 9. 回滚步骤
1. 网关切流回旧版本 Web/API/WS。
2. 停止新版本实例。
3. 若涉及迁移：执行 `down` 回滚脚本（仅允许可逆迁移）。
4. 恢复旧版本配置。
5. 验证核心链路（登录、会话、工单、AI、配置）。
6. 写入 `RELEASE_ROLLBACK` 审计记录。
