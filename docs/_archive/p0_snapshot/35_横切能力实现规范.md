## 2. 横切能力范围（P0）
- 多租户隔离
- 权限守卫与数据范围
- 审计日志
- 幂等控制
- 统一异常与错误码
- AI 限流、重试、降级
- request_id 链路追踪

## 3. 多租户隔离实现

### 3.1 强制规则
- 所有业务查询默认附加 `org_id` 条件。
- Repository 层禁止暴露“无 org 过滤”查询方法。
- 任何跨租户访问返回 `AUTH_FORBIDDEN`。

### 3.2 建议实现
- NestJS `TenantContextGuard`：从 JWT 解析 `org_id` 注入 request context。
- Repository 基类封装 `withOrgScope(query, orgId)`。

## 4. 权限与数据范围实现

### 4.1 双层校验
1. Controller 使用 `@RequirePerm(PERM-XXX)`。
2. Service 再次校验数据范围（self/team/org）。

### 4.2 会话权限强制细分
- 发送：`PERM-CNV-SEND`
- 接入：`PERM-CNV-ACCEPT`
- 转接：`PERM-CNV-TRANSFER`
- 关闭：`PERM-CNV-CLOSE`

禁止聚合权限替代。

## 5. 审计拦截器实现

### 5.1 触发范围
- 全部写接口（POST/PUT/DELETE/状态流转）。
- 敏感读接口（审计查询、权限配置查询）。

### 5.2 审计字段
`request_id/org_id/operator_id/api_id/action/target_type/target_id/result/error_code/before_data/after_data`。

### 5.3 审计失败处理
- 业务成功但审计失败：记录错误日志并告警（不回滚业务）。
- 审计服务不可用超过阈值：触发 P1 告警。

## 6. 幂等中间件实现

### 6.1 适用接口
- 详见 `11_API` 第 7 章（需要 `Idempotency-Key` 的接口）。

### 6.2 行为
1. 检查 `Idempotency-Key`。
2. 读取 Redis 键（`org_id:api_id:key`）。
3. 命中则直接返回缓存响应。
4. 未命中则执行业务并写入缓存（TTL=10m）。

## 7. 异常与错误码映射
- 领域异常 -> 业务错误码：
  - 状态机异常 -> `STATUS_TRANSITION_INVALID`
  - 并发冲突 -> `CONFLICT_VERSION`
  - 权限拒绝 -> `AUTH_FORBIDDEN`
- 统一异常过滤器确保响应结构一致。

## 8. AI 横切策略
- 限流：`org_id + user_id` 维度。
- 重试：仅网络/5xx 错误重试 1 次。
- 熔断：连续失败触发，30s 后半开。
- 降级：失败后返回可人工接管提示。

## 9. request_id 与日志规范
- 网关入站生成 `request_id`（若客户端未传）。
- 全链路透传 `request_id` 至 API、WS、Audit。
- 日志最小字段：`ts/level/request_id/org_id/user_id/api_id/cost_ms/code`。

## 10. 测试要求
- 跨租户查询拦截。
- 缺权限按钮 API 拦截。
- 幂等重复提交返回一致结果。
- 审计字段完整性。
- 异常码映射一致性。
