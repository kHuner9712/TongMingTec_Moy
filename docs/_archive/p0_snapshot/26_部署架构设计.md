## 2. 部署目标
- 支撑 P0 DEV/TEST/PROD 全链路运行。
- 确保 API、WebSocket、数据库、缓存、前端协同可观测、可回滚。

## 3. 服务拓扑
```
Internet
  -> LB / API Gateway
      -> Web (React)
      -> API (NestJS HTTP)
      -> WS Gateway (NestJS WebSocket)
           -> Redis (pub/sub, cache, idempotency)
           -> PostgreSQL (业务数据 + 审计)
```

## 4. 组件职责与依赖
| 组件 | 职责 | 关键依赖 |
| --- | --- | --- |
| Web | 页面渲染、API 调用、WS 消费 | API Gateway、WS Gateway |
| API | 业务 API、鉴权、状态机、审计 | PostgreSQL、Redis |
| WS Gateway | 实时事件推送、订阅鉴权 | Redis、PostgreSQL（必要时补拉） |
| PostgreSQL | 主数据存储 | - |
| Redis | 缓存、幂等键、限流、WS 分发 | - |

## 5. 环境划分
| 环境 | 用途 | 数据要求 |
| --- | --- | --- |
| DEV | 开发联调 | 可重置数据 |
| TEST | 集成测试 | 接近生产，保留回归样本 |
| PROD | 线上 | 高可用、备份与监控 |

## 6. 网络与安全
- 全链路 HTTPS/WSS。
- API 与 WS 统一 JWT 鉴权。
- 数据库与 Redis 仅内网访问。
- 密钥从环境变量/密钥管理系统注入，不写入代码仓库。

## 7. 健康检查接口
| 服务 | 路径 | 通过条件 |
| --- | --- | --- |
| API | `/api/v1/health/live` | 进程存活 |
| API | `/api/v1/health/ready` | PostgreSQL + Redis 可用 |
| WS | `/ws/health` | 进程存活 + Redis 连接正常 |
| Web | `/health` | 静态资源可访问 |

## 8. 可观测性要求
- 日志：结构化 JSON（含 `request_id/org_id/user_id/api_id/latency_ms/result_code`）。
- 指标：
  - API QPS、P95、5xx 比例
  - WS 连接数、重连率、事件投递延迟
  - PostgreSQL 连接池利用率
  - Redis 命中率与内存使用
- 告警：见 `24_实施与上线指南.md`。

## 9. 高可用与容灾
- PostgreSQL：主从复制 + 每日备份。
- Redis：持久化 + 主从。
- API/WS：至少 2 实例（PROD）。
- Web：多实例 + CDN（可选）。
